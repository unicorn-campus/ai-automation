## 슬라이드 1 — 비용(Cost) 하네스 적용 방안

**상단 제목**: 무한 루프·토큰 누수·Agent 폭주를 호출 경계의 결정론적 통제로 예방한다

**핵심 관점**: Cost 리스크는 실행이 길어지거나, 컨텍스트가 비대해지거나, Agent 호출이 통제되지 않을 때 커짐.  
하네스는 호출·완료 경계에서 결정론적으로 강제(✅)하되, 생성 도중 제어가 필요한 항목(△)은 보완 관행과 함께 설계함.

### 컬럼 ① 무한 루프 (실행이 끝나지 않아 과금 누적)

- ✅ 해결책:
  - 최대 반복 횟수 (Stop/SubagentStop 완료 게이트)
  - 반복 패턴 감지 (PreToolUse: 동일 tool_input 해시 비교 후 deny)
  - 부분 결과 반환 (Stop: 미완 시에만 block)
- Skill 적용 예시: 웹 검색 Skill은 최대 재시도를 2회로 제한하고, 같은 쿼리가 반복되면 즉시 종료해 마지막 성공 결과를 반환함.
- Agent 적용 예시: 리서치 Agent는 계획→검색→요약 루프를 최대 3회로 제한하고, 동일 도구 호출이 반복되면 Stop 게이트가 중단한 뒤 현재까지 요약·로그를 남김.
- △ 일부 적용: 없음 — ✅로 전수 통제

### 컬럼 ② 토큰 누수 (컨텍스트 비대화로 토큰 과금 증가)

- ✅ 해결책:
  - 입력 절삭 (PreToolUse updatedInput으로 과대 입력 절삭)
  - 핸드오프 요약 전달 (서브에이전트가 최종 요약만 반환)
  - 요청별 사용량 측정 (PostToolUse(Agent) totalTokens·usage 적재)
- Skill 적용 예시: 문서 요약 Skill은 원문 전체 대신 핵심 단락만 입력받고, 결과는 5줄 요약과 핵심 키워드만 반환하도록 입력·출력을 설계함.
- Agent 적용 예시: 상담 Agent는 전체 대화 이력 대신 요약 메모리만 다음 단계로 넘기고, 요청별 입력·출력 토큰을 PostToolUse로 기록해 모니터링함.
- △ 일부 적용:
  - 이력 요약 (PreCompact로 압축 직전 핵심 이력 보존·아카이브 — 요약 품질 강제는 불가) → 보완: 요약 프롬프트 품질 관리

### 컬럼 ③ Agent 폭주 (Agent 호출 미통제로 비용 급증)

- ✅ 해결책:
  - 호출 수 제한 (고비용 도구 카운터 + 상한 초과 시 PreToolUse deny)
- Skill 적용 예시: 번역 Skill은 1회 요청당 외부 API 호출 1회, 병렬 작업 2개까지만 허용하고, 호출 카운터가 상한을 넘으면 PreToolUse가 deny함.
- Agent 적용 예시: 여행 추천 Agent는 하위 Agent에 Agent 도구를 주지 않아 호출 깊이를 제한하고, 예산 초과·호출 급증 시 Kill-switch 플래그로 고비용 도구를 차단한 뒤 간략 응답으로 전환함.
- △ 일부 적용 (호출 경계 게이트만 가능, 생성 중 토큰 절단 불가 — 보완 병행):
  - Budget 한도 (PreToolUse deny): 누적 초과 시 후속 호출만 차단 → 보완 예산 대시보드·알람
  - Kill-switch (PreToolUse deny): 폭주 플래그 시 고비용 도구 일괄 차단 → 보완 운영자 수동 중단 절차
  - 동시 실행 제한 (flock 카운터): in-flight 동시성 근사만 가능 → 보완 오케스트레이션 큐
  - 깊이 제한 (tools에서 Agent 제거): 서브에이전트 호출 차단 → 보완 Agent 설계 시 깊이 명시

**설계 포인트**:
- Skill 설계 = 입력·출력·토큰·재시도 한도를 명확히 정의(PreToolUse/Stop 게이트)
- Agent 설계 = 예산·호출 수·중단 조건·로그를 함께 설계(PostToolUse 적재 + PreToolUse deny)
- 운영 원칙 = 비용은 사후 정산이 아니라 호출 경계의 결정론적 통제, 생성 중 통제는 SDK·운영으로 보완

**제외(❌)**: 단계별 타임아웃·컨텍스트 하드캡 (wall-clock 타임아웃·런타임 자동 압축 — SDK/워치독 영역)

## 슬라이드 2 — 성능(Performance) 하네스 적용 방안

**상단 제목**: 장애 전파·품질 저하·지연을 도구·완료 경계의 차단으로 막는다

**핵심 관점**: 성능 리스크는 실패가 연쇄될 때, 검증 없는 출력이 흐를 때, 직렬·중복 실행으로 지연될 때 커짐.  
하네스는 도구·완료 경계에서 실패와 품질을 차단(✅)하되, wall-clock·전송 계층은 SDK/인프라에 위임함.

### 컬럼 ① 장애 전파 (한 단계 실패가 전체로 번져 시스템 마비)

- ✅ 해결책:
  - Circuit Breaker (PostToolUseFailure 실패 카운트 + 임계 초과 시 일정시간 deny)
- Skill 적용 예시: 외부 API 조회 Skill은 연속 실패가 임계치를 넘으면 일정 시간 호출을 차단하고, 캐시된 직전 결과로 대체 응답함.
- Agent 적용 예시: 데이터 수집 Agent는 고부하·위험 작업을 서브에이전트로 격리해 메인 컨텍스트 오염을 막고, 실패 시 대체 경로 지침을 주입받아 계속 진행함.
- △ 일부 적용:
  - 리소스 격리 (위험 작업을 서브에이전트로 컨텍스트 격리 — 인프라 CPU·메모리 격리는 불가) → 보완: 컨테이너 배포
  - Graceful Degradation (PostToolUseFailure + additionalContext로 대체 응답 지침 주입 — 폴백 내용 강제는 soft) → 보완: 폴백 응답 템플릿 표준화

### 컬럼 ② 품질 저하 (검증 없는 잘못된 출력이 다음 단계로 전파)

- ✅ 해결책:
  - 검증 게이트 3종:
  - 스키마 검증 (출력 JSON 스키마 위반 시 PostToolUse/Stop block)
  - 교차 검증 (다중 검증자 비교 후 불일치 시 차단)
  - 검증 실패 차단 (근거·테스트 검증 실패 시 다음 단계 block)
- Skill 적용 예시: 분류 Skill은 결과를 고정 JSON 스키마로 받고, 스키마 위반 시 Stop hook가 block해 잘못된 출력이 후속 단계로 넘어가지 못하게 함.
- Agent 적용 예시: 분석 Agent는 두 검증자 Agent의 결과를 비교해 불일치하면 차단하고, 근거·테스트 검증을 통과하지 못한 출력은 다음 단계로 넘기지 않음.
- △ 일부 적용:
  - Grounding (근거·출처 존재 검증 후 누락 시 block — 정확성 강제는 soft) → 보완: 사람 사실검증·인용 검수

### 컬럼 ③ 지연·낭비 (직렬 실행·중복 조회로 응답 지연)

- ✅ 해결책:
  - 병렬 실행 (독립 읽기전용 조회 readOnlyHint를 병렬 호출)
- Skill 적용 예시: 멀티소스 조회 Skill은 서로 독립인 읽기 작업을 병렬로 호출하고, 동일 질의 결과를 MCP 캐시에 저장해 재사용함.
- Agent 적용 예시: 리서치 Agent는 독립 검색을 병렬로 수행해 대기시간을 줄이고, 자주 쓰는 조회 결과는 캐시에서 먼저 확인함.
- △ 일부 적용:
  - 캐싱 (MCP·스크립트로 조회 결과 캐시·재사용 — 프롬프트 캐시 자동 제어는 불가) → 보완: 캐시 TTL·무효화 정책

**설계 포인트**:
- Skill 설계 = 출력 스키마·근거 검증 게이트를 명확히 정의(PostToolUse/Stop block)
- Agent 설계 = 실패 차단·교차 검증·리소스 격리를 함께 설계(PostToolUseFailure + 서브에이전트)
- 운영 원칙 = 성능은 사후 튜닝이 아니라 실패·품질을 경계에서 즉시 차단, 타임아웃은 인프라로 보완

**제외(❌)**: 계층적 타임아웃·스트리밍 응답·Backpressure (wall-clock·전송·인프라 계층 — SDK/게이트웨이 영역)

## 슬라이드 3 — 보안(Security) 하네스 적용 방안

**상단 제목**: 프롬프트 인젝션·데이터 유출·권한 오남용을 입력·전송·도구 경계에서 차단한다

**핵심 관점**: 보안 리스크는 외부 입력이 에이전트를 조종할 때, 민감정보가 유출될 때, 고위험 도구가 무통제로 실행될 때 커짐.  
하네스는 입력·전송·도구 경계에서 결정론적으로 차단(✅)하되, 프로세스 격리는 OS·인프라에 위임함.

### 컬럼 ① 프롬프트 인젝션 (외부 입력·문서가 에이전트 행동을 조종)

- ✅ 해결책:
  - 입력 필터링 (UserPromptSubmit에서 악성 패턴 block)
  - 불신 데이터 표시 (외부 본문을 신뢰/불신 컨텍스트로 분리·태그)
- Skill 적용 예시: 웹 본문 요약 Skill은 가져온 외부 텍스트를 불신 태그로 감싸 지시문과 분리하고, 악성 지시 패턴은 UserPromptSubmit에서 차단함.
- Agent 적용 예시: 메일 처리 Agent는 외부 메일 본문을 불신 컨텍스트로 표시해 명령으로 해석하지 않고, 신뢰 지시와 분리해 처리함.
- △ 일부 적용:
  - 역할 분리 (신뢰/불신 컨텍스트 분리, 사실형 컨텍스트 주입 — 주입 저항 강제는 soft) → 보완: 시스템 프롬프트 가드·정기 레드팀

### 컬럼 ② 데이터 유출 (민감정보가 외부로 전송·노출)

- ✅ 해결책:
  - DLP 필터 (전송 입력에 주민·카드번호 정규식 탐지 시 PreToolUse deny)
  - 민감정보 마스킹 (updatedInput 전송 전·updatedToolOutput 표시 전 치환)
  - 감사 로그 (PostToolUse로 모든 도구 호출 전수 기록)
- Skill 적용 예시: 외부 전송 Skill은 전송 직전 주민·카드번호를 탐지하면 deny하고, 통과 데이터도 마스킹 후 전송하며 모든 호출을 감사 로그에 남김.
- Agent 적용 예시: 고객 응대 Agent는 외부 API 전송 전 DLP 필터로 민감정보를 차단하고, 허용 도메인 외 호출은 네트워크 화이트리스트로 막음.
- △ 일부 적용:
  - 네트워크 화이트리스트 (WebFetch·Bash(curl) URL 화이트리스트 외 deny — 패킷 레벨·난독화 차단은 불가) → 보완: 단일 egress MCP 경유 시 ✅로 승격

### 컬럼 ③ 권한 오남용 (고위험 도구가 통제 없이 실행)

- ✅ 해결책:
  - 최소 권한 (tools·disallowedTools로 필요 도구만 허용)
  - 도구 화이트리스트 (허용 외 도구·인자 deny, 권한 모드 우회 불가)
  - HITL 승인 (고위험 작업에 PreToolUse ask 게이트)
  - 시간·작업 기반 권한 (date·작업 분류 후 조건부 deny/ask)
- Skill 적용 예시: 파일 정리 Skill은 삭제·외부전송 도구를 disallowedTools로 빼고, 위험 작업은 PreToolUse ask로 사람 승인을 받음.
- Agent 적용 예시: 운영 Agent는 업무시간 외·대량 삭제를 시간·작업 분류로 deny/ask하고, 최소 권한으로 필요한 도구만 부여받음.
- △ 일부 적용: 없음 — ✅로 전수 통제

**설계 포인트**:
- Skill 설계 = 최소 권한·도구 화이트리스트·DLP를 명확히 정의(PreToolUse deny/ask)
- Agent 설계 = 불신 데이터 분리·감사 로그·HITL 승인을 함께 설계(UserPromptSubmit + PostToolUse)
- 운영 원칙 = 보안은 사후 탐지가 아니라 입력·전송·도구 경계의 사전 차단, 격리는 인프라로 보완

**제외(❌)**: OS 샌드박스 (deny는 접근 차단일 뿐 프로세스·리소스 격리가 아님 — 컨테이너·격리 VM 영역)
