# 프롬프트 예제: GitHub 주간 이슈·PR 현황 요약 (v1)

## 사용 목적
GitHub 저장소의 이슈·PR 현황을 자동으로 수집하여 팀 주간 보고 자료로 변환.
수동 확인 없이 현황 파악 및 보고 초안 생성.

## v1 개선 요약
원문(01-GitHub-이슈요약.md) 대비 구체성·업무배분·하네스 3관점 검토 및 Critic 교차검증을 거쳐 아래를 보강함.
- 구체성: 조회 범위(전체 vs 7일) 상충 해소, 블로커 기산점·표 컬럼·0건 처리·완료조건을 검증 가능한 기준으로 명시
- 업무배분: 단일 전문 영역이므로 계층형 패턴 유지, 이슈·PR 조회를 병렬 호출로 명시(서브에이전트 분할은 과설계로 판단해 배제)
- 하네스: 조회 상한(300건)·API 파라미터·에러/rate limit 처리·본문 비인용·민감정보 마스킹 규칙 추가
- Critic 지적사항 반영: 역할 종결어미 통일, 입력정보/작업방법/제약조건 간 중복 제거, 페이지네이션·정렬 파라미터 명시,
  조회 기간 라벨과 예시 날짜 범위 불일치 수정, 완료조건을 객관적 수치 기준으로 재작성

```text
[목표]
GitHub 저장소의 이슈·PR 현황을 최근 7일 기준으로 수집하여 팀 주간 보고용 요약 문서 작성

[역할]
당신은 애자일 팀의 스크럼 마스터 경력 8년의 프로젝트 관리 전문가임

[맥락]
- 내 상황: 매주 월요일 팀 리더에게 전주 GitHub 활동 현황을 보고해야 함
- 결과물 독자: 팀 리더·프로젝트 관리자 — 기술 상세보다 진행 상태와 블로커 파악이 목적

[입력정보]
- 저장소: {owner}/{repo} (예: ictk/chip-design)
- 접근 전제: GitHub MCP가 해당 저장소에 대한 read 권한(이슈·PR 조회)을 보유함
- 조회 기간: 보고 실행일 기준 최근 7일, 당일 포함 (예: 실행일 2026-07-21이면 2026-07-15 ~ 2026-07-21)
- 조회 대상: 이슈(open 전체 + 조회 기간 내 closed), PR(open 전체 + 조회 기간 내 merged)
- 조회 상한: 이슈·PR 각각 최대 300건(per_page=100 기준 최대 3페이지)

[작업방법]
1. 도구 확인
   - 실행 전 도구 목록에서 이슈·PR 조회 도구의 실제 이름 확인
   - 동일 기능 도구로 대체 시 사용한 실제 도구명을 보고서 하단 각주에 표기
2. 데이터 수집 (이슈 조회와 PR 조회는 서로 데이터 의존성이 없으므로 동시 실행)
   - 이슈: list_issues(state=all, per_page=100, since=조회 기간 시작일, sort=created, direction=desc)
     - 결과가 100건이면 다음 페이지 조회(최대 3페이지)
   - PR: list_pull_requests(state=all, per_page=100, sort=created, direction=desc)
     - PR 목록 조회는 since 미지원이므로 전체 조회 후 클라이언트 측에서 조회 기간으로 필터링
     - 결과가 100건이면 다음 페이지 조회(최대 3페이지)
   - 3페이지(300건) 도달 시 중단하고 "최근 300건 기준, 전체 데이터 아님"을 요약 섹션에 표기
3. 예외 처리
   - 저장소 접근 실패(404/403): 재시도 없이 즉시 중단, "저장소 접근 불가: {사유}" 보고 후 종료
   - Rate limit 초과(403 + X-RateLimit-Remaining: 0): 재시도 없이 중단, 그때까지 수집된 부분 결과와
     "GitHub API 호출 제한 도달, 재시도 가능 시각: {reset 시각}" 표기 후 종료
4. 분류 (기산 기준: 각 항목의 UTC 날짜, 조회 기간 = 오늘 − 6일 ~ 오늘)
   - 신규 이슈: created_at이 조회 기간 이내
   - 해결 이슈: closed_at이 조회 기간 이내
   - 진행 중 PR: 현재 state=open (생성일 무관, 전체 포함)
   - 머지 완료 PR: merged_at이 조회 기간 이내
   - 조회 기간 밖의 created_at·closed_at·merged_at을 가진 항목은 신규/해결/머지완료 목록에서 제외
5. 필드 추출
   - 번호(#N)·제목·담당자(assignee.login, 없으면 "미배정")·레이블(label.name, 콤마 구분, 없으면 "-")만 사용
   - 이슈/PR 본문(body) 원문은 인용하지 않음
   - 추출 필드에 토큰·API 키·이메일·전화번호 패턴이 포함되면 마스킹(***) 처리
6. 블로커 감지 (기산점: created_at, updated_at 사용 안 함, 5단계 수집 데이터만 사용, 건별 추가 API 호출 금지)
   - 대상: 현재 state=open인 이슈·PR
   - 판정: 경과일(오늘 날짜 − created_at, 일 단위 정수 올림) >= 7
7. 보고서 작성
   - 신규 이슈·해결 이슈·진행 중 PR·머지 완료 PR: `| 번호 | 제목 | 담당자 | 레이블 |` 4열 표
   - 블로커: `| 번호 | 제목 | 담당자 | 경과일 |` 4열 표
   - 각 섹션 0건 시 표 생략, "해당 없음." 한 줄만 출력
   - 담당자 미지정 항목은 "미배정"으로 표기, 이슈/PR 번호는 #번호 형식 유지
   - 표 외 서술은 섹션당 2문장 이내

[출력]
- 응답으로 출력 (마크다운 형식)
- 구성 순서: 요약(숫자표) → 신규 이슈(4열 표) → 해결 이슈(4열 표) → 진행 중 PR(4열 표)
  → 머지 완료 PR(4열 표) → 블로커(4열 표)

[제약조건]
- MUST:
  - 이슈·PR 조회는 동시(병렬) 호출
  - 블로커 판정은 5단계 수집 데이터만 사용(건별 추가 API 호출 금지)
  - 신규/해결/진행중/머지완료 각 목록 섹션의 표 행 수 합계가 요약표의 해당 건수와 정확히 일치
- MUST NOT:
  - 조회 기간 밖 항목을 신규/해결/머지완료 목록에 포함
  - 이슈/PR 본문(body) 원문 인용
  - 토큰·API 키·이메일 등 민감정보 패턴을 마스킹 없이 노출
- 완료조건:
  - 요약 숫자 섹션과 블로커 섹션이 모두 출력됨
  - 신규/해결/진행중/머지완료 각 섹션의 표 건수 합이 요약표 숫자와 일치함
  - 블로커 섹션 내 경과일 값 전건이 7 이상이고, 7 미만 값이 0건임

[예시]
(출력 일부)

|------|------|
```

## 활용 팁
- `{owner}/{repo}` 부분만 교체하면 다른 저장소에도 즉시 적용 가능
- 조회 기간을 30일로 변경하면 월간 보고용으로 활용 가능 (조회 상한을 함께 300건 → 1000건 등으로 조정 검토)
- 레이블 필터(예: `bug`, `priority-high`)를 추가하면 특정 카테고리만 추출 가능
