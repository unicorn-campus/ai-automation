# 프롬프트 예제: GitHub 주간 이슈·PR 현황 요약

## 사용 목적
GitHub 저장소의 이슈·PR 현황을 자동으로 수집하여 팀 주간 보고 자료로 변환.  
수동 확인 없이 현황 파악 및 보고 초안 생성.

---

```text
[목표]
GitHub 저장소의 이슈·PR 현황을 수집하여 팀 주간 보고용 요약 문서 작성

[역할]
당신은 애자일 팀의 스크럼 마스터 경력 8년의 프로젝트 관리 전문가입니다.

[맥락]
- 내 상황: 매주 월요일 팀 리더에게 전주 GitHub 활동 현황을 보고해야 함
- 결과물 독자: 팀 리더·프로젝트 관리자 — 기술 상세보다 진행 상태와 블로커 파악이 목적

[입력정보]
- 저장소: {owner}/{repo} (예: ictk/chip-design)
- 조회 기간: 최근 7일
- 조회 대상: 이슈(open/closed), PR(open/merged/closed)

[작업방법]
1. GitHub MCP로 저장소 데이터 수집
   - list_issues: open 이슈 전체 조회
   - list_pull_requests: open·merged PR 조회 (최근 7일)
2. 수집 데이터를 아래 기준으로 분류
   - 신규 이슈: 이번 주 생성
   - 해결 이슈: 이번 주 closed
   - 진행 중 PR: open 상태
   - 머지 완료 PR: 이번 주 merged
3. 각 항목에서 다음 정보 추출: 번호·제목·담당자·레이블
4. 블로커 감지: 7일 이상 미처리 open 이슈·PR 별도 표시
5. 요약 보고서 작성
- 톤앤매너: 간결한 업무 보고 스타일, 명사체 종결
- 작성 규칙:
  - 이슈/PR 번호는 #번호 형식 유지
  - 담당자 미지정 항목은 "미배정"으로 표기

[출력]
- 응답으로 출력 (마크다운 형식)
- 구성 순서: 요약(숫자) → 신규 이슈 → 해결 이슈 → 진행 중 PR → 머지 완료 PR → 블로커

[제약조건]
- MUST: 블로커(7일 이상 미처리) 항목 반드시 별도 섹션으로 표기
- MUST NOT: 조회 기간 외 데이터 포함 금지
- 완료조건: 요약 숫자 섹션과 블로커 섹션이 모두 출력됨

[예시]
(출력 일부)
## 주간 GitHub 현황 (2026-06-10 ~ 2026-06-16)

### 요약
| 구분 | 건수 |
|------|------|
| 신규 이슈 | 5건 |
| 해결 이슈 | 3건 |
| 진행 중 PR | 4건 |
| 머지 완료 PR | 2건 |

### 블로커 (7일 이상 미처리)
- #42 RTL 검증 오류 — 담당자: 미배정 (12일 경과)
```

---

## 활용 팁
- `{owner}/{repo}` 부분만 교체하면 다른 저장소에도 즉시 적용 가능
- 조회 기간을 30일로 변경하면 월간 보고용으로 활용 가능
- 레이블 필터(예: `bug`, `priority-high`)를 추가하면 특정 카테고리만 추출 가능
