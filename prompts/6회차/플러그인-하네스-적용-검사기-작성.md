[목표]
비용·성능·보안 하네스 통제 중 플러그인으로 적용 가능한 항목만으로 구성된, 개발된 Claude Code
플러그인을 진단하는 재사용 검사기 문서 「harness-checker.md」를 작성함.

[역할]
당신은 MAS(멀티에이전트 시스템) 운영 리스크 통제 7년 경력의 시니어 하네스 엔지니어이며,
Claude Code 플러그인·Hook(PreToolUse·PostToolUse·Stop 등) 설계와 비용·성능·보안 게이트
구현을 전문으로 함.

[맥락]
- 내 상황: ICTK 업무 자동화 플러그인을 배포하기 전후로 9대 실행 리스크를 표준 절차로 점검할
  도구가 필요함. 이번 세션에서 하네스 통제별 "플러그인 적용 가능성(✅/△/❌)"을 공식 문서 근거로
  검증 완료함.
- 결과물 독자: 플러그인을 개발한 개발자(점검 수행자). Claude Code skill·agent·plugin·hook
  기초를 이해함.
- 활용 방식: 본 프롬프트로 검사기 문서를 1회 생성함. 이후 플러그인을 만들 때마다 이 문서를 열어,
  ① 각 항목의 적용 가능성(✅/△/❌)을 자기 플러그인·버전 기준으로 재판정하고, ② 체크리스트로
  현재 적용 상태(Y/N)를 근거(파일:라인)와 함께 평가하며, ③ 미적용 항목을 구현 방법대로 보완해
  하네스 적용 방안을 도출함. 산출물은 1회성 보고서가 아니라 반복 사용하는 진단 도구임.
- 두 개의 축 구분: "적용 가능성(✅/△/❌)"은 플랫폼·플러그인 구조가 결정하는 구현 가능 여부이고,
  "적용 여부(Y/N)"는 이 플러그인의 현재 상태임. 전자는 재판정 규칙으로 갱신, 후자는 매번 평가함.

- Hook 구현 방법: `references/harness-hooks.md`
  (이벤트·핸들러 타입·차단 신호·복붙 JSON, 모든 Fact에 출처 절 표기)
- 하네스 적용 가능성 기준선(이번 세션 검증, 2026-06-21, `harness-hooks.md` 기준): 아래 XML
  태그 내용을 검사기의 초기 체크 항목 선별 기준이자 재판정의 출발점(baseline)으로 사용함.

<하네스적용가능성기준선>
판정: ✅ 적용 가능(결정론적 강제) / △ 조건부(호출·도구 경계만, 외부 상태 조립 또는 soft) /
      ❌ 구조적 불가(런타임·SDK·OS·인프라 필요, 플러그인 범위 밖)

[비용]
무한 루프
- ✅ 최대 반복 횟수 — 서브에이전트 maxTurns + Stop hook 8회 block cap
- ✅ 반복 패턴 감지 — command hook이 tool_input 비교 + 상태파일
- ✅ 부분 결과 반환 — Stop hook 완료 게이트(stop_hook_active 확인)
- ❌ 단계별 타임아웃 — 생성·도구 wall-clock 타임아웃 hook 없음
토큰 누수
- ✅ 입력 절삭 — PreToolUse updatedInput
- ✅ 핸드오프 요약 전달 — 서브에이전트 최종 블록만 반환
- ✅ 요청별 사용량 측정(서브에이전트 단위) — PostToolUse(Agent) totalTokens·usage 텔레메트리
- △ 이력 요약 — PreCompact·절삭 가능, 요약 품질 강제는 soft
- ❌ 컨텍스트 하드 윈도 캡 — 런타임 자동 압축 전속
- ❌ 메인 에이전트 실시간 토큰 측정 — hook에 실시간 미제공
Agent 폭주
- ✅ 호출 수 제한 — PostToolUse 카운트(상태파일) + PreToolUse deny
- △ Budget 한도 — 호출 경계 누적비용 게이트(deny), 생성 중단 불가
- △ Kill-switch — deny=추가 작업 차단, 진행 중 토큰 절단 불가
- △ 동시 실행 제한 — flock 카운터 + 경계 deny 근사, 완전 강제 아님
- △ 깊이 제한 — 추가 스폰 차단 가능, 임의 숫자 깊이는 불가(런타임 고정)

[성능]
시스템 마비
- ✅ Circuit Breaker — PostToolUseFailure 실패 카운트 + PreToolUse deny
- △ Graceful Degradation — PostToolUseFailure+additionalContext 복구 지침, 폴백 내용 soft
- △ 리소스 격리 — 컨텍스트 격리(서브에이전트)만, 인프라 격리 불가
- ❌ Backpressure — 인바운드 큐·부하 차단은 세션 밖 인프라
응답 지연
- ✅ 병렬 실행 — 병렬 도구·서브에이전트
- △ 캐싱 — MCP·스크립트로 직접 구축(프롬프트 캐시는 런타임 자동)
- ❌ 계층적 타임아웃 — 에이전트·다계층 wall-clock 타임아웃 없음
- ❌ 스트리밍 응답 — 전송 계층, 서브에이전트 핸드오프는 단일 블록
할루시네이션 전파
- ✅ 스키마 검증 — Stop·PostToolUse hook 또는 스크립트 검증 후 block
- ✅ 교차 검증 — 멀티에이전트 구조
- ✅ 검증 실패 차단 — hook block·게이트
- △ Grounding — 근거 존재 검증 가능, 정확성은 soft

[보안]
Prompt Injection
- ✅ 입력 필터링 — UserPromptSubmit hook block
- ✅ 불신 데이터 표시 — 신뢰·불신 컨텍스트 분리·태그
- △ 역할 분리 — 표시 가능, 주입 저항은 soft
- ❌ OS 샌드박스 — deny는 접근 차단일 뿐 프로세스·리소스 격리는 OS 영역
데이터 유출
- ✅ DLP 필터 — PreToolUse 정규식 스캔 + deny
- ✅ 민감정보 마스킹 — PreToolUse updatedInput·PostToolUse updatedToolOutput
- ✅ 감사 로그 — PostToolUse hook 전수 기록
- △ 네트워크 화이트리스트 — 도구 deny 가능, 패킷 레벨·Bash 난독화 차단 불가
권한 오남용
- ✅ 최소 권한 — tools·disallowedTools
- ✅ 도구 화이트리스트 — tools·permissions
- ✅ HITL 승인 — PreToolUse ask(권한 모드 우회 불가)
- ✅ 시간·작업 기반 권한 제한 — command hook date·작업 분류 + deny·ask

중요 한계(검사 시 반영)
- updatedToolOutput은 Claude가 보는 내용만 변경 — 실제 전송 차단은 PreToolUse 필수
- if 필터는 best-effort(fail-open) — 하드 강제는 permission system 병행
- 차단은 exit 2(exit 1은 비차단), agent hook은 실험적·고비용(프로덕션은 command 권장)
