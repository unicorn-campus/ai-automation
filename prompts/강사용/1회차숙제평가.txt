[작업방법]
1. 제출물 수집
   - references/gdrive-collect-guide.md 절차대로 사용자 폴더를 재귀 탐색하여 제출 파일을 찾고 내용을 읽음
   - 콘티·튜터·데브·스펙이 사용자 폴더 범위를 나누어 병렬 수집(무한루프 방지: 방문 폴더 ID 관리)
   - 산출: homework/1회차/수집매니페스트.md
     - 표: 사용자명|소유자(이메일)|요약|회의록|시장조사|추가과제|제출수|비고(구조/노이즈)
     - 각 셀 근거로 fileId·modifiedTime을 함께 보관(상세표 또는 주석)
     - 빈 폴더=미제출, 권한오류=별도 표기(미제출로 단정 금지)

2. 프롬프트 취합 파일 작성 (과제별)
   - 산출: homework/1회차/prompt-{과제}.md (요약/회의록/시장조사/추가과제 각 1개)
   - 사용자명을 섹션 제목으로, 제출 프롬프트 원문을 코드블록으로 기재

3. 부정제출 탐지 (수집 매니페스트 기반, 과제별 수행 — 오탐 방지 최우선)
   - 신호 A (수강생 간 복제): 과제별로 제출물을 짝지어 비교
     - 섹션 헤더·공백·마크다운 이스케이프를 제거한 '고유 내용'(페르소나·규칙·예시·업무맥락)만 비교
     - 유사도는 재현 가능하게 계산: 정규화 텍스트에 difflib.SequenceMatcher 비율 또는 토큰 자카드
       (Bash로 python 스크립트 실행 가능)
     - 고유내용 유사도 >= 0.85 → 복제 의심. 공통 골격만 유사한 경우는 플래그 금지
   - 신호 B (예제/보일러플레이트 복붙): 강사 제공 예제·AGENTS.md 기본 골격과 실질 동일하고
     본인 업무·도메인 개인화가 없으면 성의 없는 제출로 표시
   - 신호 C (메타데이터 이상: get_file_metadata의 owner/createdTime/modifiedTime)
     - 서로 다른 owner가 동일 내용 제출(계정 공유·전달 복제)
     - 한 사용자의 모든 파일이 createdTime≈modifiedTime이고 상호 수 초 이내(반복 개선 흔적 없는 일괄 업로드) — 보조 신호
   - 판정 (2단계, 자동 확정 금지)
     - 의심(검토대상): 신호 1개 → Best 3 선정 보류, 강사 수동 확인 요청. 0점 아님
     - 부정 확정: 신호 2개 이상 + 강사 확인 → 0점 처리
   - 증거 필수: 각 플래그에 (대상자, 과제, 유사도 수치, 일치 구간 2~3줄 인용, 메타데이터 근거) 기록
   - 산출: homework/1회차/부정제출-검토.md (의심/확정 구분, 근거 포함)

4. 과제 평가 (클로니·스펙·콘티·튜터·데브 5인 병렬, 각자 독립 채점)
   - 평가 기준(가중치)과 앵커드 루브릭 — 각 기준 100점 만점, 아래 수준 서술어로 채점
     - ① 구조 완성도 (25%): 8대 섹션 충족 + 섹션 정합성(한 섹션 한 정보, 혼입 없음)
       - 0: 섹션 구분 없음 / 40: 필수 섹션 다수 누락 / 60: 필수 섹션 존재하나 혼입·형식오류 다수
         / 80: 필수 섹션 완비, 경미한 흠 / 100: 8대 섹션 완비 + 정합성 우수
     - ② 명확성·검증가능성 (25%): 모호어 없이 측정·검증 가능, 판정기준·완료조건 객관성, 입출력 경로 특정
       - 0: 모호·추상 일색 / 40: 지시는 있으나 모호어 다수 / 60: 일부 구체적, 완료조건 불명
         / 80: 대체로 구체·검증 가능 / 100: 전 항목 측정 가능, 완료조건이 객관 증거 기준
     - ③ 결과 품질=실효성 (35%): 기준 시나리오에서 예상되는 산출물의 유용성·정확성·충실성(실행하지 않고 설계 기반 예측)
       - 0: 산출 불가로 예측 / 40: 요구 미충족 예상 / 60: 요구 일부 충족 예상
         / 80: 요구 충족·실무 사용 가능 예상 / 100: 요구 초과·즉시 실무 적용 수준 예상
     - ④ 추가과제 품질 (15%): 본인 업무 프롬프트의 실무 적합성·구체성 (제출만으로 만점 아님)
       - 0: 미제출 / 40: 제출했으나 일반적·피상적 / 60: 업무 맥락 반영
         / 80: 구체적·실행 가능 / 100: 즉시 업무 투입 가능한 완성도
   - 결과 품질(③) 예상 평가 방법 (프롬프트를 실제 실행하지 않음)
     - [입력정보]의 <예상평가시나리오>를 입력했다고 가정하고, 산출될 결과물을 설계 기반으로 예측하여 채점
     - 예측 근거로 아래 설계 신호를 인용: 산출 형식·스키마 규정의 구체성 / 지시·판정기준의 명확성 /
       예외·환각·불확실성 처리 규칙 유무 / 입력→출력 매핑의 누락 위험
     - 전원 동일 기준 시나리오로 예측하여 공정 비교. 실측이 아니므로 점수 옆에 '예상' 표기
     - 판단이 갈리면 보수적으로 낮은 쪽을 택하고 근거를 남김
   - 종합점수 = ①x0.25 + ②x0.25 + ③x0.35 + ④x0.15 (100점 만점)
   - 팀원별 결과표: homework/1회차/평가수행결과표-{팀원}.md
     - 표: 수강자명|구조(25%)|명확성(25%)|결과품질(35%)|추가과제(15%)|종합점수|평가의견
     - 평가의견은 각 기준 점수의 근거를 구체 인용과 함께 기술(수강자가 납득 가능하게)
     - 부정 확정자는 0점, 평가의견에 근거 기술. 의심(검토대상)은 점수 산정하되 '보류' 표시

5. 평가 종합
   - 평가자 간 종합: 기준별로 5인 점수 중 최고·최저 1개씩 제외한 절사평균 사용(편향 완화)
   - 편차 플래그: 특정 수강자·기준에서 평가자 간 점수차 20점 이상이면 '재검토' 표시
   - 산출: homework/1회차/평가종합표.md
     - Best Top3: 이름·종합점수·선정사유(강점 2~3개를 원문 인용과 함께)
     - 전체순위: 등수|수강자명|구조|명확성|결과품질|추가과제|종합점수
     - 사용자별 종합 평가의견
     - 부정제출: 확정/의심 구분과 근거

6. 완료보고
   - 산출 파일 목록과 각 파일의 실제 생성 여부를 확인하여 보고(정직한 보고 규칙 준수)
