# 업무자동화후보-정보요청서

**ICTK 업무자동화 후보 정보 요청서**

업무 1건당 1장 · 약 5분 · AI 지식 불필요

**0. 한 장 요약 (이 문서 사용법)**

이 문서는 안내·작성요령(1~3절) → 빈 양식(4절) → 작성 예시(5절) 순서임. 4절 양식만 채우면 되고, 처음이면 5절 예시를 먼저 보면 됨.

**1. 안내 (왜 받고, 어떻게 쓰이나)**

본 요청서는 ICTK 조직 구성원이 지금 손으로 반복하는 업무 중 'AI가 대신해 주면 좋을 일'을 찾기 위한 양식임.

AI·프로그래밍 지식은 전혀 필요 없으며, 평소 동료에게 일을 설명하듯 쉬운 말로 작성함. 


**보안 안내 (보안 IC 기업 필수 준수)**

· 영업비밀(PUF·암호 IP 설계값·키), 미공개 경영·IR 정보, 고객 개인정보, 고객 NDA·기밀의 실제 값은 적지 않음 — '어떤 종류인지'만 기술

· 예시 자료를 첨부할 때는 민감 부분을 가린(마스킹) 샘플로만 첨부

· 마스킹조차 어려운 민감 업무는 파일 대신 결과물의 항목·구조만 글로 적고 실제 값은 적지 않음

· 본 요청서는 사내 보안 경로로만 제출

**2. 작성 요령 — 좋은 후보 vs 나쁜 후보**

작성 전 내 업무가 자동화에 적합한지 가볍게 자가 점검함. 애매하면 일단 적고 팀이 판단하게 함.

**자동화에 잘 맞는 업무 (이런 신호면 적극 제출)**

· 입력 → 처리 → 출력 흐름이 한 방향으로 명확하고 매번 거의 같은 순서로 반복됨

· 판단 기준이 규칙으로 설명됨(예: '10만원 이상은 빨강'). 규칙이 없으면 「항상 같음」으로 명시 가능

· 입력·결과물이 디지털 자료(엑셀·PDF·메일·시스템 화면)라 기계가 읽고 쓸 수 있음

· 수행 빈도가 잦고 1회 처리량·소요시간이 커서 절감 효과(ROI)가 큼

· 실제 입력 → 출력 예시 한 벌을 마스킹해 첨부할 수 있어 결과 형식을 고정 가능

· 사람이 확인할 단계를 구분할 수 있어 안전한 자동화 승인 지점이 분명함

**자동화가 어려운 업무 (이런 신호면 효과 낮음 — 무리하지 않음)**

· '알아서 잘 정리'처럼 결과 기준이 사람의 감각·재량에 달려 매번 달라짐

· 단계가 매번 바뀌거나 예외가 대부분이라 규칙으로 설명 불가

· 입력이 종이·구두·손글씨·머릿속 암묵지뿐이라 디지털 재료가 없음

· 1년에 1~2번 하는 일이라 자동화 투입 대비 효과가 낮음

· 고도의 전문 판단·창작·대외 협상이 핵심이라 사람이 최종 책임져야 하는 일

· 영업비밀·미공개 IR 등 민감도 최상인데 마스킹·격리가 불가능한 업무

· 예시·설명이 추상적이어서 무엇을 만드는지 팀이 재현할 수 없음

**3. 작성 항목 한눈에 보기**

본 요청서는 기본 정보(메타) 7칸 + 업무 내용(핵심) 8칸으로 구성됨.

**4. 요청서 양식 (이 절을 작성)**

**4-1. 기본 정보 (메타 7칸)**

**4-2. 업무 내용 (핵심 8칸)**

각 칸의 안내를 참고하고 예시처럼 작성함. 회색 작성란에 기재.









**5. 작성 예시 (현업 담당자가 실제로 채운 1건)**

품질·신뢰성·보안인증 조직 담당자가 작성한 가상 예시임. 비개발 담당자가 자기 말로 적은 톤을 그대로 보여줌 — 이 정도 구체성이면 팀이 변환 가능함.

**업무: 고객 반품(RMA) 접수 → 불량분석 보고서 초안 자동작성**

| ④ 어떤 순서로 처리하나요 (★ = 사람 확인 필수) |
| --- |
| 1. 반품 메일·첨부를 다 열어 증상·반품 수량·고객사·칩 번호를 파악 2. 첨부 측정값 엑셀에서 어떤 항목이 규격을 벗어났는지 확인 3. 사내 생산이력 시스템에서 칩 번호로 LOT·웨이퍼·출하검사 수치를 조회해 옮김 4. 과거 비슷한 불량 보고서가 있었는지 폴더에서 찾아봄 5. 보고서 앞부분(담당팀·문제 내용·임시 조치·불량 현상)까지 채움 ★6. 원인 분석과 재발 방지 대책은 내가 직접 판단해 적음 ★7. 품질팀장 검토 후 고객 회신 |


| ⑤ 경우마다 달라지는 판단 규칙 |
| --- |
| 반품 수량 10개 이상이거나 같은 LOT에서 두 번째 들어온 반품이면 '중대'로 분류해 팀장에게 즉시 보고(빨강 표시). 고객이 NDA 맺은 곳이면 보고서에 그 고객의 최종 제품명은 안 적고 '고객 application'으로만 기재. 증상이 '동작 안 함(무반응)'이면 정전기(ESD) 의심 항목을 점검표에 추가, '간헐 오류'면 온도·전압 조건 항목을 추가. 측정값이 규격 안에 드는데도 반품된 경우는 '재현 안 됨' 후보로 따로 표시. 원인 칸은 어떤 경우든 사람이 정하므로 항상 비워두고 후보만 나열 |


| ⑥ 결과물은 무엇인가요 (형태 · 받는 사람 · 나가는 곳) |
| --- |
| 워드로 된 불량분석 보고서 1건. 1차 독자는 품질팀장(사내 검토용), 검토·확정 후 고객사 품질담당자에게 전달. 나가는 곳: ☑ 고객에게 전달(외부). 따라서 사내 기밀(LOT·웨이퍼 번호 등)을 가린 고객용 버전이 별도로 필요 |


| ⑦ 결과물이 맞는지 보는 기준 (검수 포인트) |
| --- |
| · 고객사 이름·반품 수량·증상이 고객 메일과 정확히 일치 · 측정값·단위·칩 번호는 원자료와 100% 일치(틀리면 사고) · LOT·웨이퍼 번호가 생산이력 시스템 값과 일치 · 원인 분석·재발 방지 칸은 비어 있어야 함 — AI가 채우면 안 되고 후보 목록만 · 고객용 최종본에 LOT·웨이퍼·담당자 개인정보가 남아 있으면 안 됨(마스킹 확인) |
