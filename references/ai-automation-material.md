## 1. 세션 01 — Claude Code 입문 및 프롬프트 기초

### 1.1 WHY — 왜 업무 자동화인가

- 반복 작업은 AI에게, 사람은 판단에 집중함. 자동화의 목적은 "사람을 대체"가 아니라 "사람이 더 중요한 일에 집중"하게 만드는 것임.
- 역할 분담: 사람="무엇을·왜"를 정함(판단) → AI="어떻게"를 빠르게 실행(작업) → 사람=결과를 최종 확인·승인(게이트).

### 1.2 Claude Code란? (HOW)

- 공식 정의: 코드베이스를 읽고, 파일을 편집하고, 명령을 실행하며, 개발 도구와 통합되는 도구임.
- 쉬운 정의: "내 폴더의 파일을 읽고 정리해 주는, 대화로 일시키는 AI 비서".
- 데스크톱 앱(Anthropic 제공)의 **3개 탭** 중 'Code' 탭이 곧 Claude Code임.

| 탭 | 역할 |
|----|------|
| Chat | 파일 접근 없는 일반 대화 (claude.ai와 유사) |
| Cowork | 내 PC의 격리된 가상 환경에서 작업 — 비개발자의 AI 비서 목적 |
| **Code** | 내 PC에서 직접 작업 — 파일을 직접 다루는 코딩 보조 (← 본 교육에서 사용) |

### 1.3 사용법·환경

- 작업 머신: **로컬**(PC에서 작업) / **클라우드**(Anthropic VM에서 작업, Git 저장소 연결 후 소스 다운로드).
- 대화창: 줄바꿈은 SHIFT-ENTER, 실행은 ENTER.
- 수행 모드: 권한 요청 / 계획 모드 / 편집 수락 / 권한 건너뛰기 / 자동 모드.
- 사이드 패널: 미리보기(로컬 웹 실행) / Diff(변경 사항) / 터미널 / 파일 / 작업 / 계획.
- 설치 전제: 유료 계정(Pro·Max·Team·Enterprise) 필요. 무료 claude.ai 플랜으로는 'Code' 탭 사용 불가.

### 1.4 기본 명령

| 명령 | 설명 |
|------|------|
| `/goal` | 목표(완료 조건)를 달성할 때까지 턴을 넘어 끝까지 작업 (장기 작업 무중단) |
| `/remote-control` | 핸드폰·다른 머신에서 현재 작업을 원격 제어 (URL 발급) |

### 1.5 중요 설정 파일

**시스템 프롬프트** (프롬프트 수행 시마다 자동 로딩, 지켜야 할 지침·규정 기록)

| 경로 | 적용 범위 |
|------|----------|
| `~/.claude/CLAUDE.md` | 현재 OS 유저 전체에 적용 |
| `{디렉토리}/CLAUDE.md` | 현재 디렉토리에만 적용 |
| `{디렉토리}/.claude/rules/*.md` | CLAUDE.md를 목적별로 분할할 때 사용 |
| `{디렉토리}/AGENTS.md` | AI 코딩툴 메모리 표준 파일. CLAUDE.md에는 `@AGENTS.md`만 기록하여 사용 가능 |

**프로젝트 메모리** (프로젝트별 장기 기억 저장소)

- `~/.claude/projects/{project dir}/MEMORY.md` — 프롬프트에 "auto-memory에 기록 + {내용}"으로 지시.
- autoDream: 사용자가 Claude를 사용하지 않을 때 지난 24시간의 프로젝트 메모리를 자동 정리하는 기능.

### 1.6 작업 폴더·권한·안전

- 작동 경계: Claude는 선택한 폴더 안에서만 작업함. 읽기(프로젝트 폴더·하위 인식), 쓰기(그 폴더 범위 안에서만).
- 사람 최종 승인 게이트: 기본 모드=**권한 요청**. 변경마다 Accept(수락)/Reject(거절), 수락 전에는 파일이 바뀌지 않음.
- '권한 건너뛰기'는 사람 승인 없이 수정/삭제가 가능하므로 안전한 경우에만 선택함.

### 1.7 프롬프트 엔지니어링 — 5대 구성요소

- 황금률: "이 업무를 잘 모르는 동료에게 그대로 시켜보라. 그가 헷갈리면 AI도 헷갈린다."
- Claude를 "똑똑하지만 우리 회사 규칙을 전혀 모르는 신입 직원"처럼 대함.

| 요소 | 핵심 내용 |
|------|-----------|
| ① 목적 | 요청의 목적을 명확하게 제시 |
| ② 역할/맥락 | AI에게 역할과 맥락(상황·관점)을 제시 |
| ③ 입력·처리·출력 | 입력(참고자료/외부 서치) · 처리(작업 요구사항·단계) · 출력(형식·파일 경로) |
| ④ 제약조건 | MUST(필수) · MUST NOT(금지) · 완료조건(체크리스트) |
| ⑤ 예시(few-shot) | 결과 예시 내용 또는 예시 파일 경로 |

- 제1원칙: "사람에게 요청하듯 명확하고 구체적으로 작성". 응답을 보며 AI와 함께 프롬프트를 계속 발전시키는 것이 중요함.
- 작성 흐름: Human이 초안 작성(5요소별 섹션) → Claude에게 검토/개선 요청 → 수행 → 결과를 보고 함께 추가 개선.
- 프롬프트는 파일로 저장하여 관리하면 재활용·공유에 유리함.

### 1.8 실습

- 실습1: 핵심 3줄 요약 / 실습2: 표준 회의록 / 실습3: 시장 조사.
- 공통 안전 수칙: 민감·대외비 자료는 사용하지 않음(일반·예시 자료로), 결과는 사람이 최종 확인 후 사용.

## 2. 세션 02 — 커넥터/루틴/Computer use

### 2.1 WHY — "말"에서 "연결·자동화"로

- 1회차는 프롬프트로 "말"만 함(사람이 매번 데이터 복붙·실행). 업무 자동화를 위해 Claude가 도구에 직접 접근 필요.
- HOW: ① 외부 도구 연결(커넥터) ② 정해진 시각에 자동 반복(루틴) ③ 화면 직접 조작(Computer use).

### 2.2 커넥터·MCP

| 용어 | 정의 |
|------|------|
| 커넥터(Connector) | 클릭만으로 외부 도구를 잇는 통로 (코드·터미널 불필요, 그래픽 설정 흐름) |
| MCP | 통로의 **기술 표준 이름**. 공식 비유 "AI를 위한 USB-C 포트" |
| MCP 서버 | 연결되는 도구 하나하나(단위). Claude에 도구·데이터 접근 제공 (예: GitHub, 지도, Gmail) |

- 연결 흐름(GUI): '사용자 지정' → 커넥터 둘러보기 → 커넥터 선택 → [연결] → OAuth 승인 → 권한 설정(그룹별 항상 허용/승인 필요/차단).
- OAuth: 보안을 강화한 인증 방식이자 MCP의 표준 인증 방식. 토큰은 Claude 서버에서 안전하게 관리·자동 갱신됨.
- PlayMCP(playmcp.kakao.com): 카카오가 제공하는 MCP 연결 통로(카카오 인증 필요).

### 2.3 루틴(Routine) — 정해진 시각에 자동 실행

- 정의: 프롬프트 + (저장소) + 커넥터를 한 번 묶어, 정해진 시각에 자동으로 실행하는 단위.

| 유형 | 선택 기준 | 특징 |
|------|----------|------|
| **Remote** (클라우드 루틴) | 컴퓨터를 꺼도 자동으로 돌아야 할 때 | 실행 중 승인 프롬프트 없이 자율 실행 (예: 주간 PR·백로그 요약) |
| **Local** (데스크톱 예약 작업) | 내 PC 안의 파일을 직접 다뤄야 할 때 | 작업별 권한 모드를 따름 (예: 아침 캘린더·받은편지함 브리핑) |

### 2.4 Computer use — 화면을 보고 직접 조작

- 정의: Claude가 화면을 스크린샷으로 보고 마우스·키보드로 GUI를 조작함(사람이 하듯 앱을 다룸).
- 동작(반복 순환): ① 화면을 본다(스크린샷) ② 클릭·타이핑 ③ 앱 실행 ④ 결과 확인 → 반복.
- 꼭 기억할 점: (1) 사람처럼 화면을 봄 (2) 느려서 마지막 수단 — 빠르고 정밀한 커넥터가 먼저 (3) 위험한 건 사람이(로그인·결제·약관 동의), Esc로 즉시 중단.
- 현재 연구 프리뷰, 기본 OFF, Pro·Max 전용(Team·Enterprise 불가). 진짜 데스크톱에서 동작하므로 고위험 조작은 데모·관찰만.

### 2.5 안전 수칙 — 자동화 기본기 3원칙

| 원칙 | 내용 |
|------|------|
| 1. 데이터 격리 | 선택한 폴더 안에서만 작업. 추가 폴더는 명시 허용해야 접근. 클라우드 루틴은 선택 저장소·커넥터 범위로 한정 |
| 2. 최소 권한 | 연결 전 서버 신뢰 여부 확인. 불필요한 커넥터 제거, Computer use는 앱별로 좁게 승인 |
| 3. 사람 최종 승인 | 수행 모드를 '권한 요청'으로. 로그인·결제·발급·제출 등 고위험은 사람이 직접 수행 |

- 프롬프트 인젝션: 외부 문서·화면 속 숨은 지시문이 Claude를 속여 엉뚱한 동작을 시키는 공격 — 신뢰되지 않은 서버 연결을 경계함.

## 3. 세션 03 — 스킬 개발 기초

### 3.1 WHY / HOW / WHAT

- WHY: 같은 지시·체크리스트·여러 단계 절차를 채팅에 반복해서 붙여넣고 있다면, 바로 그때가 Skill로 만들 시점임.
- HOW: 그 절차를 SKILL.md 파일 하나에 자연어로 적어두면 Claude가 자기 "도구함"에 추가함.
- WHAT: 이후엔 자동 트리거(요청이 맞으면) 또는 `/{스킬이름}`으로 바로 재사용함.

### 3.2 Skill이란?

- 정의: SKILL.md에 지침을 적어 Claude의 능력을 넓히는 것. 비유는 "요리 레시피"(한 번 적어두면 같은 결과를 빠르게 재현).
- 3가지 효과: 전문가화(Specialize) / 반복 제거(Reduce repetition) / 조합(Compose — 여러 스킬을 이어 복잡한 흐름 구성).

### 3.3 SKILL.md 구조

- 위에서 아래로 읽는 구조: ① 설정칸(Formatter, 언제 쓸지) = `name`·`description` → ② 본문(어떻게 할지) = 1회차 5요소의 확장.
- 꼭 필요한 것은 **description**(무엇 + 언제). 본문은 짧게(권장 500줄 이하), `name`은 옵션.
- 한 번 불려오면 대화 내내 남아 길수록 매번 비용이 듦.

| 옵션 | 설명 |
|------|------|
| `disable-model-invocation` | true 설정 시 자동 호출 차단, `/` 명령으로만 호출 (Default: false) |
| `user-invocable` | false 설정 시 `/` 명령 호출 차단 (Default: true) |
| `allowed-tools` / `disallowed-tools` | 사용 가능 / 불가능한 툴 리스트 (미지정 시 모든 도구 허용) |

- Tools 리스트: 파일작업(Read/Write/Edit/NotebookEdit), 검색(Glob/Grep), 실행(Bash), 웹(WebSearch/WebFetch),
  오케스트레이션(Agent), 실행(Skill/AskUserQuestion/TaskCreate/TaskUpdate/ToolSearch), MCP(`mcp__<server>__<action>`).

### 3.4 자동 호출 원리 — description이 핵심

- 흐름: ① 시작 시 이름·description만 인지 ② 내 요청과 비교 ③ 맞으면 SKILL.md 본문을 불러와 실행.
- 좋은 description 원칙: 무엇 + 언제를 사용자 키워드로, 3인칭으로 작성("엑셀 파일을 처리하고 보고서를 만듦" O / "제가 도와드릴게요" X), 모호한 표현 금지.

### 3.5 스코프(저장 위치 = 사용 범위)

| 스코프 | 저장 위치 | 적용 범위 |
|--------|----------|----------|
| 유저(개인) | `~/.claude/skills/이름/SKILL.md` | 내 모든 폴더에서 사용 |
| 프로젝트 | `.claude/skills/이름/SKILL.md` | 그 작업 폴더에서만 |

- 호출 이름은 폴더 이름에서 옴(예: `summarize-changes/` → `/summarize-changes`). 만들거나 고치면 보통 재시작 없이 즉시 반영됨.

### 3.6 내장(Built-in) 스킬

| 스킬 | 용도 |
|------|------|
| `/goal` | 완료 조건을 선언하면 충족될 때까지 끝까지 작업 |
| `/code-review` | 변경한 코드의 버그·정리할 곳 점검 (수행 전 git commit 필요) |
| `/security-review` | 변경분을 보안 취약점(인젝션·인증·데이터 노출) 관점에서 분석 (commit 필요) |
| `/simplify` | 코드 다듬기 — 재사용·단순화·효율·추상화 수준 4관점 자동 검토 |

### 3.7 안전 수칙 — 스킬도 "소프트웨어처럼"

1. 신뢰 출처만: 내가 만들었거나 Anthropic 제공한 것만. 외부 스킬은 SKILL.md·스크립트를 전부 감사 후 사용.
2. 민감 정보 그대로 적지 않기: 비밀번호·고객 정보·핵심 보안 자산을 SKILL.md에 적지 않음(파일은 공유·커밋될 수 있음).
3. 사람 최종 승인: 배포·제출처럼 되돌리기 어려운 작업은 자동 호출을 끄고 사람이 직접 호출.

## 4. 세션 04 — 스킬 심화 / 웹 MCP / MCP 개발

### 4.1 WHY — Skill에 "손과 눈"을 달면

- 3회차 Skill은 "지침" + "기존 커넥터"로 제한됨(웹브라우저 조작·커넥터에 없는 기능은 못 씀).
- MCP로 웹브라우저를 조작하고 필요한 기능을 직접 개발하여 스킬에 꽂아줌.

### 4.2 웹 자동화 2종

| 구분 | Playwright MCP | Claude in Chrome |
|------|----------------|------------------|
| 브라우저 | 별도 자동화 브라우저 창 | 현재 사용 중인 브라우저의 새 탭 |
| 로그인 상태 | 분리(깨끗) — 내 세션과 별개 | 공유 — 이미 로그인한 사이트 접근 |
| 잘 맞는 일 | 공개 웹 정보 조회·정리 | 로그인이 필요한 내 계정 작업 |
| 동작 방식 | 페이지 구조(접근성 트리)를 읽어 클릭·입력·이동 (빠르고 안정적) | 로그인·CAPTCHA 시 멈추고 사람에게 처리 요청 |
| 설치 | Playwright MCP 설치 필요 | 설정에서 활성화 필요 |

- CAPTCHA: 컴퓨터와 인간을 구분하기 위한 자동화된 테스트("나는 로봇이 아닙니다" 등).
- 도우미 MCP 2종 — context7(라이브러리 최신 공식 문서를 실시간 조회, "use context7"), sequential thinking(복잡한 문제를 단계적 사고로 분해).

### 4.3 MCP 개요

- 정의: AI 앱이 외부 도구를 호출하는 **표준 프로토콜**(= 표준 인터페이스 = 규약).
- 필요성: 앱-도구 간 인터페이스 표준화 / 프레임워크 비종속 / 풍부한 생태계(사실상 업계 표준).

### 4.4 MCP 핵심 기능(Primitives)

| 기능 | 용도 | 호출 의사결정 | 실제 호출 |
|------|------|--------------|-----------|
| **Tools** | 외부 서비스 실행 (API 호출, DB 변경, 메일 발송) | LLM | AI 앱 |
| **Resources** | 서버가 가진 데이터 참조 (설정값, 스키마, 이력 조회) | AI 앱 | AI 앱 |
| **Prompts** | 반복 작업의 표준화된 LLM 프롬프트 템플릿 (코드 리뷰, 버그 분석) | 사용자/AI 앱 | AI 앱 |

- 4단계 생명주기: ① 초기화 요청/응답 ② 초기화 완료 알림 ③ 작업 수행(목록 조회·실행) ④ 종료.
- FastMCP(MCP 개발 라이브러리): `@mcp.tool()`/`@mcp.resource()`/`@mcp.prompt()` 데코레이터 + 타입 힌트 + docstring으로 JSON Schema 자동 생성.
- 고급 기능: Sampling(LLM 텍스트 생성, Host가 검토/승인/거부) / Roots(작업 루트 디렉터리 권고) / Elicitation(스키마 기반 UI로 추가 정보 끌어내기).

### 4.5 MCP 아키텍처·통신·인증

- 아키텍처: **Host**(MCP Client를 구동하는 머신) / **Client**(MCP Server와 통신하는 앱, 예: Claude Code) /  
  **Server**(Tools·Resources·Prompts 제공).

| 통신 방식 | 설명 |
|-----------|------|
| **STDIO** | MCP Host가 서브 프로세스로 직접 호출. Client와 동일 머신에 존재 |
| **HTTP** | MCP Client가 HTTP로 호출. 보통 Client와 다른 머신에 존재 |

- 인증/인가: 표준은 OAuth. Public Client(로컬: 사용자 PC, Claude Desktop·IDE) /  
  Credential Client(서버: VM·Container, 백엔드·배치). PKCE 적용.
- OAuth·JWT 인증은 **HTTP 방식일 때만 필요**함. STDIO 방식은 Host가 서브 프로세스로 직접 실행하므로 불필요.
- 본 교육에서는 원격 MCP 제작은 다루지 않고 Local MCP를 개발하여 연동하는 방법만 학습함. 코드는 Claude가 작성·설명하므로 비개발자도 가능.

## 5. 세션 05 — 에이젼트 개발

### 5.1 WHY — 혼자 다 하면 빠뜨리고, 나눠 맡기면 꼼꼼해진다

- 한 명이 글도 쓰고 검토도 하면 자기 실수를 못 봄. 작성과 점검을 다른 역할이 맡으면 품질이 올라감.
- 일을 잘게 나눠 '작성자'·'검토자' 같은 서브에이전트에게 맡기고, 메인이 결과를 종합함.

### 5.2 에이전트 vs 서브에이전트

| 구분 | 정의 |
|------|------|
| 에이전트(메인=오케스트레이터, 팀장) | 스스로 여러 단계를 해내는 AI 일꾼. 일을 쪼개 누구에게 맡길지 정하고, 결과를 모아 최종 정리 |
| 서브에이전트(팀원) | 특정 역할만 맡는 보조 AI. 자기 독립 작업 공간에서 처리하고 결론(요약)만 올림 |

- 따로 두는 이유: 역할마다 도구·권한을 다르게 줄 수 있어 안전하고, 저렴한 모델에 맡겨 비용도 아낌.

### 5.3 오케스트레이션 흐름 (3단계)

1. **분배(Distribute)** — 일을 쪼개 알맞은 서브에이전트에 위임.
2. **실행(Run)** — 각자 자기 공간에서 독립 실행(안 섞임).
3. **종합(Synthesize)** — 메인이 요약을 모아 최종 정리.

- 독립적인 일은 동시에(병렬), 이어지는 일은 차례로(체인). 둘 다 결과는 메인이 모음.

### 5.4 서브에이전트 만들기

- 정의 필드: `name` · `description` · `tools` · `model`. 출력 경로 예: `~/.claude/agents/{name}.md`.
- 용도 구분: **스킬은 다단계로 구성된 여러 작업**, **에이젼트는 독립적인 작은 작업**을 하는 목적으로 만듦.
- 스킬+에이젼트 구조: 메인이 다단계를 직접 하는 것이 비용·성능·보안상 부적절할 때 전환. 이 경우 스킬·에이젼트를 **프로젝트 스콥**으로 만듦.
  (유저 스콥으로 분업 구조가 필요하면 플러그인이 해결책)
- 서브에이젼트는 백그라운드에서 수행되므로 사용자 인터랙션 작업은 메인 에이젼트가 해야 함.

| 빌트인 에이젼트 | 모델 | 목적 | 허용 도구 |
|----------------|------|------|-----------|
| Explore | Haiku(경량) | 코드베이스 검색·분석(읽기 전용) | Read, Grep, Glob |
| Plan | 주대화 상속 | 컨텍스트 수집(읽기 전용) | Read, Grep, Glob |
| General-purpose | 주대화 상속 | 복잡한 다단계 작업(탐색 + 수정) | 모든 도구 |

### 5.5 하네스 엔지니어링

- 스킬/에이젼트/플러그인 개발 시 MAS(멀티 에이전트 시스템) 9대 실행 리스크를 고려하여 **비용·성능·보안** 측면을 조율함.

## 6. 세션 06 — 플러그인 개발 및 배포

### 6.1 WHY — '나 혼자'에서 '우리 팀'으로

- 잘 만든 스킬·에이전트를 동료에게 주려면 폴더 복사·설정 설명이 필요해 공유가 번거롭고 버전이 제각각임.
- 관련 스킬·에이전트·MCP·훅을 한 폴더(Plugin)로 묶고 마켓플레이스에 올리면 클릭 한 번으로 설치됨.

### 6.2 Plugin이란?

- 정의: 스킬·에이전트·MCP·훅을 한 폴더에 묶어 공유·설치하는 패키지. 비유는 "스마트폰 앱".
- 3가지 가치: 한 번에 설치(Bundle) / 팀 공유(Share) / 버전·업데이트(Version).

### 6.3 플러그인 디렉토리 구조

| 파일/디렉토리 | 필수 | 설명 |
|---------------|------|------|
| `plugin.json` | **필수** | 플러그인 이름·버전·설명 등 기본 메타데이터 |
| `marketplace.json` | 선택 | 마켓플레이스 제출용 추가 메타데이터 |
| `skills/` | 선택 | SKILL.md 파일 디렉토리 |
| `agents/` | 선택 | 에이전트 정의 파일 디렉토리 |
| `hooks/` | 선택 | hooks.json — 이벤트 기반 자동화 |
| `.mcp.json` | 선택 | MCP 서버 구성 파일 |
| `.lsp.json`, `settings.json` | 선택 | LSP 구성 / 기본 설정 파일 |

### 6.4 라이프사이클 명령(현재 OS 사용자 전체 적용)

```
claude plugin marketplace add ./                          # 로컬 소스
claude plugin marketplace add {Git Org}/{Git Repo}        # Git 소스
claude plugin install {plugin-name}@{marketplace-name}    # 설치
claude plugin update  {plugin-name}@{marketplace-name}    # 업그레이드
claude plugin list / claude plugin marketplace list       # 조회
claude plugin remove  {plugin-name}@{marketplace-name}    # 삭제
```

- 설치 후 `/{plugin-name}:{command}`로 연결된 스킬을 호출함.
- **마켓플레이스를 삭제하면 그 마켓플레이스로 설치한 플러그인도 자동 삭제됨.**

### 6.5 Hook(훅) — 특정 순간에 자동 실행되는 규칙

| 이벤트 | 시점·용도 |
|--------|-----------|
| **PreToolUse** | 도구 실행 직전 — 위험하면 막기(차단 exit 2) 또는 마스킹 |
| **PostToolUse** | 도구 실행 직후 — 결과 점검·후처리 |
| **UserPromptSubmit** | 내가 프롬프트를 보낼 때 |
| **SessionStart** | 세션이 시작될 때 |

- 핵심: 사람이 일일이 검사하지 않아도 규칙대로 자동 작동 → 민감 정보 유출 방지에 강력. 차단뿐 아니라 마스킹(예: 주민번호 → `******`)도 가능.

### 6.6 안전 수칙 — 플러그인도 "소프트웨어 설치처럼"

1. 신뢰 출처만: 플러그인은 내 PC에서 코드를 실행할 수 있음 — 내가 만들었거나 Anthropic·사내 검증 마켓플레이스만. 설치 전 '무엇이 설치되는지' 확인.
2. 훅으로 민감 정보 마스킹: 핵심 보안 자산이 새지 않도록 훅으로 자동 차단·마스킹(사람 실수를 시스템이 보완).
3. 사람 최종 승인: 파일 생성·배포는 Accept/Reject로 확인. 버전은 검증된 번호만 올려 무단 전파 차단.
