#### 과제2
## 요약
### 잘된점
- WHY("장 시작 전 아침에 밤사이 뉴스를 한 번에 훑고 싶은 개인 투자자")가 명확해 실무 목적이 뚜렷함  
- UI·분류·데이터·렌더러를 분리하고 데이터 JSON 스키마를 예시로 못 박아 재현성이 높음  
- 완료조건에 "메일 발송 성공 로그 확인", "git push 완료"를 넣어 검증 가능함  
### 보완할점
- "새 주제면 새 id 기입"의 신규 카테고리 판정 기준이 모호함 — 구체성 보완 필요  
- 다수 출처를 병렬 수집한다면서 실제 위임 주체가 단일임 — 역할배분 보완 필요  
- 기사 우선순위·선정 개수 기준이 없어 결과가 매번 달라질 수 있음 — 구체성 보완 필요  
## 설명
### 구체성
- 잘된점: `data/YYYYMMDD_HHMM.json` 스키마를 category/title/url/summary/source/time 필드까지 명시하고  
  "200자 내 핵심 요약", "최근 24시간 뉴스만 필터링" 등 측정 가능한 기준을 제시함  
- 보완할점: "`categories.json`에 없는 새 주제면 새 id + `category_label`"의 "새 주제" 판정 기준이 모호함.  
  "기존 라벨과 의미 중복 시 재사용, 신규 주제는 예: 'ETF/펀드'처럼 명명" 식의 판정 규칙을 명시하면 재현성이 오름  
### 실용성
- 잘된점: `python generate_newsletter.py ... --send`로 렌더링·발송까지 실행 명령이 구체적이고,  
  ".env(SMTP 자격증명) 커밋/푸시 금지"로 운영 안전장치까지 반영해 바로 루틴화 가능함  
- 보완할점: "지난 24시간 기사만 병렬 수집"에서 출처가 9곳인데 수집 건수 상한이 없어 뉴스레터 분량이 들쭉날쭉함.  
  "카테고리별 최대 N건, 총 M건 이내"처럼 상한을 두면 매일 일정한 품질의 결과물을 얻음  
### 역할별 팀원 배분
- 잘된점: "국내외 증시·경제 뉴스를 선별·요약하는 금융 뉴스 큐레이터"로 역할 정체성을 명확히 규정함  
- 보완할점: "위 출처를 웹 검색으로 ... 병렬 수집"이라 했으나 수집·번역·정제·발송을 단일 주체가 전담함.  
  수집(리서처)·요약번역(작성자)·검수(리뷰어) 파이프라인으로 분해해 서브에이전트에 위임하면 병렬성과 품질검증을 살림

#### 과제3
## 요약
### 잘된점
- 6개 점검 관점과 HIGH/MEDIUM/LOW 3등급 기준을 명시해 판정이 일관됨  
- 취약점마다 "위치(파일:라인)·설명·근거(CWE/OWASP)·수정안(원본→수정안)"을 강제해 검증 가능함  
- "지어내지 않음, 추정은 '가능성'으로 구분"으로 환각을 차단하고 완료조건까지 제시함  
### 보완할점
- 의존성 취약점 점검의 판단 근거(버전·DB)가 없어 실행이 어려움 — 실용성 보완 필요  
- 단일 보안 엔지니어에 전 관점을 몰아 대규모 디렉토리에선 과부하 — 역할배분 보완 필요  
- "오래된/취약한 의존성"의 '오래된' 기준이 모호함 — 구체성 보완 필요  
## 설명
### 구체성
- 잘된점: "HIGH: 즉시 악용 가능하고 영향이 큼 → 즉시 수정"처럼 등급 정의가 명확하고,  
  "근거: 가능하면 CWE/OWASP 항목 표기 (예: CWE-798, OWASP A02)"로 판정 근거를 구체화함  
- 보완할점: "오래된/취약한 의존성"의 "오래된" 기준이 모호함.  
  "최신 안정 버전 대비 major 2개 이상 차이 또는 알려진 CVE 존재 시 지적"처럼 판정선을 두면 재현성이 오름  
### 실용성
- 잘된점: 출력 구성을 "요약 표 → 등급별 상세 → 점검 범위" 순으로 고정하고 예시까지 제시해  
  개발자가 "리포트를 보고 수정 우선순위를 정함"이라는 목적을 바로 달성함  
- 보완할점: "오래된/취약한 의존성" 점검은 lock파일·버전 정보 없이는 판단 불가함.  
  "requirements.txt/package-lock.json 등 의존성 파일을 함께 입력받아 대조"라는 입력 요건을 명시하면 실무 적용성이 오름  
### 역할별 팀원 배분
- 잘된점: "시큐어코딩·애플리케이션 보안 점검 경력 10년의 보안 엔지니어"로 작업 성격에 맞는 전문성을 배정함  
- 보완할점: 본질적으로 단일 점검 작업이라 위임 부재는 큰 흠은 아니나, "src/" 같은 디렉토리 입력 시 단일 주체 과부하.  
  파일/관점 단위로 병렬 점검 후 오케스트레이터가 등급 취합하는 협업 패턴을 옵션으로 두면 대규모 대상에 확장됨
