## 상세 피드백

#### 과제1
## 요약
### 잘된점
- "목적지 반경 500m 내", "상위 3곳", 4열 표 등 산출물 범위·형식을 수치로 특정함  
- "조회되지 않은 소요시간·요금·영업시간을 임의로 추정·창작하지 않음" 등 정직성 제약이 명확함  
- 오케스트레이터·길잡이·맛집이·체크로 경로/맛집 병렬 조회 후 검증 게이트를 배치함  
### 보완할점
- "거리순 또는 평점순으로 정렬"은 정렬 기준이 택일로 모호하므로 1개로 확정 필요함(구체성)  
- 정렬에 쓰는 "평점"이 출력 4열(순위·장소명·거리·영업시간)에 없어 기준·출력이 불일치함(구체성)  
- 프롬프트 본문 [작업방법]은 단일 흐름 서술이라 AGENTS.md의 병렬 위임과 연결이 약함(역할배분)  
## 설명
### 구체성
- 잘된점: "카카오맵 MCP로 목적지 반경 500m 내 ... 상위 3곳을 선별"과 "순위·장소명·거리·영업시간 4열 표"로 산출물이 검증 가능하게 특정됨  
- 보완할점: "거리순 또는 평점순으로 정렬 후 상위 3곳"은 판정기준이 택일로 남아 결과가 흔들림. "거리순 우선, 동률 시 평점순"처럼 순서를 확정하고, 평점을 쓰면 출력 표에 평점 열도 추가할 것  
### 실용성
- 잘된점: "경로 안내와 맛집 추천을 한 번의 요청으로 통합 제공하기 위함"으로 WHY가 분명하고, 입력 템플릿 기반이라 재사용 가능함  
- 보완할점: "방문 인원이 많으면(4명 이상) 단체석·좌석 규모 정보가 조회되는 경우 우선 반영"은 조회 실패 시 동작이 불명확함. 미조회 시 어떻게 순위에 반영/제외할지 규칙을 명시하면 실무 재현성이 높아짐  
### 역할별 팀원 배분
- 잘된점: "경로 조회와 맛집 검색을 각 전문 팀원에게 병렬 배분"과 "정확성 검증가의 실제 도구 조회 결과 대조를 통과해야 최종 출력에 반영"으로 파이프라인+검증 게이트가 작업 구조에 적합함  
- 보완할점: 프롬프트 원문 [작업방법]에는 팀원 위임 언급 없이 단일 주체 흐름으로만 서술됨. 각 단계에 "길잡이 수행/맛집이 수행/체크 검증"을 명기해 AGENTS.md 역할과 본문을 일치시킬 것

#### 과제2
## 요약
### 잘된점
- 절대경로 인터프리터·JSON 스키마(`[{title,summary,source,url}]`)·산출 파일명까지 실행 단위로 특정함  
- "자격증명이 플레이스홀더면 ... dry-run으로 강등 ... '미검증' 종료"로 허위 완료를 차단함  
- 뉴스 큐레이터·발송 담당가·검증가로 수집→발송→대조의 파이프라인이 명확함  
### 보완할점
- "상위 3~5건을 선별"은 개수만 있고 정렬 우선순위(최신순/중요도순)가 미명시임(구체성)  
- "각 기사를 1~2문장으로 요약"의 품질 판정기준(핵심 포함 여부 등)이 없어 검증이 어려움(구체성)  
- 뉴스 수집이 단일 흐름이라 병렬 여지는 적으나, 다중 키워드 검색 분담 여지는 언급 없음(역할배분)  
## 설명
### 구체성
- 잘된점: "절대경로 인터프리터로 렌더링을 실행함(bare `python`은 Store 스텁이라 금지)"와 `output/articles_{YYYYMMDD}.json` 스키마 지정으로 실행이 재현 가능함  
- 보완할점: "상위 3~5건을 선별"은 정렬 기준이 빠져 선별 결과가 달라짐. "중요도(주요 매체·공식 발표) 우선, 동순위 최신순"처럼 정렬 규칙을 넣고 요약 품질 체크(사건·주체·결과 포함)를 완료조건에  
  추가할 것  
### 실용성
- 잘된점: "여러 매체를 직접 순회하지 않고도 ... 이메일 하나로 파악"으로 WHY가 명확하고, `.env`·스크립트 기반이라 매일 재실행 가능한 실무 자동화임  
- 보완할점: "이메일 발송 성공 응답(SMTP 전송 결과)을 확인함"은 좋으나, 발송 실패 시 재시도/알림 처리 규칙이 없음. 실패 시 로그 명시 후 종료 등 예외 흐름을 정의하면 운영 안정성이 올라감  
### 역할별 팀원 배분
- 잘된점: "정확성 검증가의 실제 발송 로그 대조를 통과해야 완료로 인정"으로 검증 게이트가 발송 자동화에 적합하게 배치됨  
- 보완할점: 작업이 대체로 순차라 위임 부재 감점은 최소이나, 다중 IT/AI 키워드 검색을 큐레이터 단일 주체에 몰았음. 키워드군별 병렬 수집 후 통합하도록 분해하면 커버리지가 넓어짐

#### 과제3
## 요약
### 잘된점
- 발견표 컬럼(위치·심각도·유형·근거·완화안·우선순위)과 [확정]/[추정] 구분으로 산출물이 검증 가능함  
- "출하 후 결함 발견 시 회수·재인증 비용이 큼"으로 WHY가 강하고 FIPS 140-3·CC 요구 연계가 실무적임  
- 키리뷰·타임가드·오버플로우·심각도로 점검 영역을 전문성에 맞게 분담하고 교차 검증 게이트를 둠  
### 보완할점
- "심각도(상/중/하)·영향도를 일관 기준으로 산정"은 상/중/하 판정의 구체 기준이 없음(구체성)  
- "교차 검증 게이트를 통과"의 통과 판정(재현 1건 이상=확정 등) 정량 기준이 모호함(구체성)  
- 4개 역할이 모두 순차 서술이라 오버플로우·키리뷰의 독립 병렬 실행 명시가 없음(역할배분)  
## 설명
### 구체성
- 잘된점: "발견표 컬럼: 위치(파일:라인) · 심각도 · 유형 · 근거 · 완화안 · 우선순위"와 예시 4행으로 출력 형식이 즉시 재현 가능하게 특정됨  
- 보완할점: "심각도·영향도를 일관 기준으로 산정함"은 상/중/하 경계가 주관적임. 공격 난이도·영향 범위·자산 중요도 축으로 된 판정 기준표(예: CVSS 유사 매트릭스)를 추가해 산정을 표준화할 것  
### 실용성
- 잘된점: "PUF·PQC(ML-KEM/ML-DSA/SLH-DSA) 구현이 FIPS 140-3·Common Criteria 요구에 부합하는지 확인"으로 보안IC 배포 전 리뷰라는 고가치 문제를 정조준함  
- 보완할점: "소스는 read-only로 읽고 직접 수정하지 않음"은 명확하나, 입력 <코드> 규모 한계(파일 수/라인 상한)가 없음. 대상 범위 상한과 미달 시 처리를 넣으면 대규모 코드에서도 재현 가능함  
### 역할별 팀원 배분
- 잘된점: "각 멤버 발견을 다른 멤버 1인 이상이 재현·반증하는 교차 검증 게이트"로 토론형 검증이 코드 리뷰 구조에 적합함  
- 보완할점: 키리뷰·타임가드·오버플로우는 상호 독립 점검이라 병렬 가능하나 처리 1~3단계가 순차 서술됨. "1~3단계 병렬 수행 후 4단계 교차검증"으로 명시하면 과부하 분산이 명확해짐
