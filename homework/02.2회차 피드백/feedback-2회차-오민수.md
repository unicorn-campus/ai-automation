## 상세 피드백

#### 과제1
## 요약
### 잘된점
- WHY가 "매일 반복되는 점심 메뉴 의사결정 피로를 AI 개인화 추천으로 해결"로 명확히 특정됨  
- "반경 200m 이내", "200m 초과 업체는 후보에서 제외" 등 판정기준을 수치로 검증 가능하게 제시함  
- "curl .../actuator/health → {"status":"UP"}" 등 실행 완료 기준을 증거 기반으로 명시함  
### 보완할점
- 목표는 점심추천 서비스인데 멤버가 전원 "칩 아키텍트/RTL 설계 엔지니어" 등 반도체팀이라 과제-역할 불일치임(역할배분)  
- 서비스 개발 타스크(카카오 연동·추천 로직)를 어느 멤버가 담당하는지 위임 매핑이 전무함(역할배분)  
- 세그먼트가 5종 나열만 되고 각 세그먼트별 추천 로직 차이가 미정의라 실행 시 모호함(구체성)  
## 설명
### 구체성
- 잘된점: "200m 이내 후보가 3개 미만이면 사용자에게 알리고, 확대 검색(예: 300m) 여부를 확인"으로 예외 흐름과 분기 조건을 구체화함  
- 보완할점: "AI 추천 필터링"이라는 표현이 추천 알고리즘·기준을 특정하지 않음. "결정 피로 세그먼트는 랜덤 3개, 건강관리는 칼로리순 정렬" 식으로 판정기준을 명시하면 검증 가능해짐(구체성)  
### 실용성
- 잘된점: "연동 흐름: 사용자 요청 → 카카오맵 검색 → AI 추천 필터링 → 카카오톡 공유"로 MCP 실행 파이프라인이 재현 가능하게 정리됨  
- 보완할점: 카카오톡 공유가 "나에게 보내기(최대 200자)"로만 되어 팀 공유 등 실무 확장이 막힘. 공유 대상·다건 처리 시나리오를 추가하면 활용도가 커짐(실용성)  
### 역할별 팀원 배분
- 잘된점: 8종 역할에 프로파일·성향·경력을 부여해 전문성 자체는 구체적으로 정의함(예: "기능 검증 엔지니어 ... 기능 커버리지 100% 달성")  
- 보완할점: 정의된 역할이 모두 반도체 설계 직군이라 "런치픽" 개발(백엔드·프론트·MCP연동)과 무관함. PO/백엔드/프론트/AI추천 담당으로 재편하고 타스크별 위임을 명시해야 함(역할배분)

#### 과제2
## 요약
### 잘된점
- "사용자는 보안 반도체(HDL/보안코어) 개발자이며 최근 24시간 보안 뉴스를 매일 아침 받아보고자 한다"로 WHY·독자가 명확함  
- data JSON·categories.json·templates 분리 구조와 JSON 스키마를 제시해 산출물이 재현 가능함  
- 완료조건을 "data JSON 생성, HTML 생성, SMTP 발송(또는 Gmail 초안), git push 완료"로 검증 가능하게 특정함  
### 보완할점
- 수집을 "병렬 에이전트(general-purpose)"로만 지정해 국내/해외 등 도메인별 전문 에이전트 분화 여지를 살리지 못함(역할배분)  
- "요약은 200자 내 사용자 친화적"에서 '친화적' 기준이 정성적이라 판정이 애매함(구체성)  
- 수집 실패·중복 판정 로직이 "중복 제거"로만 서술돼 중복 판정 기준(URL/제목 동일 등)이 미정의임(구체성)## 설명  
### 구체성
- 잘된점: "24시간 이전 뉴스 포함 금지", "출처 없는 뉴스 생성 금지", "실제 동작하는 URL 필수" 등 MUST NOT을 검증 가능한 형태로 명시함  
- 보완할점: "사용자 친화적으로 쉽게, 200자 내"의 '쉽게'가 모호함. "전문용어 1회 이상 등장 시 괄호 설명 첨부" 같은 측정 가능한 규칙으로 바꾸면 좋음(구체성)  
### 실용성
- 잘된점: "--send 가 인증 실패 등으로 실패하면 ... create_draft로 초안 생성"처럼 실패 폴백을 구체화해 실무 무중단 운영이 가능함  
- 보완할점: 스케줄이 "0 9 * * *"로 고정돼 있어 재사용 시 수신자·리포지토리가 하드코딩됨(msoh.ictk@gmail.com). 변수화하면 타 사용자 재사용성이 향상됨(실용성)  
### 역할별 팀원 배분
- 잘된점: "지난 24시간 ... 보안 뉴스를 병렬 에이전트(general-purpose)로 수집"으로 병렬화 위임을 명시해 단일주체 과부하를 회피함  
- 보완할점: 수집·번역·검증을 모두 general-purpose에 몰음. "국내 소스 수집/해외 소스+번역/사실검증" 서브에이전트로 분화하면 파이프라인 협업 구조가 명확해짐(역할배분)

#### 과제3
## 요약
### 잘된점
- 핵심 목표를 "고비용 상용 EDA 툴 ... 대체·자동화 ... 라이선스 비용 절감과 개발 자립성 확보"로 가치를 명확히 특정함  
- [역할]을 "경력 30년 수석개발자 ... EDA 플로우 ... ATE 테스터 벡터 변환 실무 지식"으로 과제에 정확히 부합시킴  
- "실제 입력으로 실행하여 기대 출력(reference)과 대조. 헤더/포맷/벡터를 바이트 단위로 diff"로 검증기준을 구체화함  
### 보완할점
- 검증(step10)·설계(step6)를 단일 페르소나가 모두 수행해 리뷰어 분리 여지를 살리지 않음(역할배분)  
- "언어별 더 나은 제안 가능"의 '더 나은' 판단기준이 미정의라 제안 근거가 모호함(구체성)  
- 입력정보의 데이터가 "데이터 : 없음"으로 예시가 비어 실제 실행 예시 검증이 어려움(실용성)## 설명  
### 구체성
- 잘된점: "repeat 카운트 최대 65536 초과 시 여러 벡터로 분할", "경계만 계산하고 동일 구간은 압축(RLE)" 등 한계·기법을 수치·기법으로 특정함  
- 보완할점: 완료조건 "기대 출력과 일치(또는 차이의 원인 규명)"에서 '일치' 허용오차가 미정의임. "바이트 100% 동일 또는 허용 diff 목록 명시"로 세분화하면 판정이 명확해짐(구체성)  
### 실용성
- 잘된점: "외부 패키지 금지, 표준 라이브러리만 사용", "CentOS + Windows11 모두, OS별 실행기 제공"으로 폐쇄망 실무 제약을 그대로 반영함  
- 보완할점: "산출물 정리(워크스페이스) tool/ reference/ manual/ ..."로 재사용 구조는 좋으나, <검토요청> 샘플 입출력이 경로 플레이스홀더(/path/to/~~~.csh)뿐이라 실제  
  재현 검증이 불가함. 최소 1개 실동작 예시를 채우면 좋음(실용성)  
### 역할별 팀원 배분
- 잘된점: 단일 script 생성이라는 과제 성격에 맞게 RTL·EDA 전문 단일 페르소나를 정확히 배정함(과제 성격상 단일주체 적합)  
- 보완할점: 생성과 "검증(필수)"이 한 주체에 묶여 자기검증 편향 위험이 있음. 검증 단계를 별도 리뷰 에이전트로 분리하면 품질 판정의 독립성이 확보됨(역할배분)
