## 상세 피드백

#### 과제1
## 요약
### 잘된점
- 루티·구르미·서머리 3역할을 파이프라인으로 분해해 위임·검토·통합 흐름이 명확함  
- "맛집 5개 이상·카카오맵 링크 모두 포함" 등 완료조건을 검증 가능하게 특정함  
- "리뷰에 없는 내용은 지어내지 않고 근거 부족"으로 환각 방지 판정기준을 명시함  
### 보완할점
- "시간대별 교통혼잡도 확인 가능하면"의 조건부 표현이 완료기준을 모호하게 함  
- 맛집 순위 판정기준이 "리뷰 평가가 더 좋은 곳"으로 정성적이라 재현성이 약함  
- 서머리의 "사용자 최종 선택 유도" 단계의 확인 방식·형식이 미지정임  
## 설명
### 구체성
- 잘된점: "경로별 비교표 작성 (경로/소요시간/환승/요금/장단점 열 포함)"으로 산출물 컬럼까지 특정함  
- 보완할점: "리뷰 평가가 더 좋은 곳 하나만 남김"은 판정기준이 주관적임 — "리뷰 언급 긍정 항목 수 우선, 동점 시 리뷰 개수" 식 정량 규칙으로 바꾸면 재현 가능함  
### 실용성
- 잘된점: "가격 확인 불가 시 확인 필요로 명시"로 실무에서 바로 신뢰 가능한 안내물이 나옴  
- 보완할점: 예산·도보 허용거리 등 데이트 제약이 없어 "최적경로"의 기준이 열려 있음 — 입력정보에 1인 예산 상한·최대 도보 분을 추가하면 필터링 기준이 명확해짐  
### 역할별 팀원 배분
- 잘된점: "루티(내비게이터)·구르미(맛집 큐레이터)·서머리(안내 편집자)"로 전문성별 분해가 적절하고 목적지별 반복으로 병렬 여지도 살림  
- 보완할점: 홍대·합정 두 목적지를 구르미 단일 주체가 순차 처리함 — "목적지별 병렬 수행"을 명시하면 과부하 없이 처리 속도를 높일 수 있음

#### 과제2
## 요약
### 잘된점
- UI·분류·데이터 분리 구조와 JSON 스키마를 예시까지 제시해 산출물이 매우 구체적임  
- "categories.json 비어있어도 정상 동작" 등 초기 상태·예외 처리를 촘촘히 규정함  
- "출처 없는 뉴스 생성 금지·24시간 이전 포함 금지" 등 MUST NOT이 검증 가능함  
### 보완할점
- "당신은 AI뉴스 수집 전문가임" 단일 역할로 수집·정제·번역·발송을 한 주체에 몰았음  
- 수집 뉴스 총 건수·소스별 최소 건수가 미지정이라 산출 규모가 열려 있음  
- "에이전트 호출하여 병렬로 수집"이라 쓰지만 호출할 에이전트가 정의되지 않음  
## 설명
### 구체성
- 잘된점: "summary: 200자 내 핵심 요약", "source: URL이 아니라 사이트명으로 표기"로 필드 규칙을 정량·명시함  
- 보완할점: "지난 24시간 내에 올라온 뉴스를 병렬로 수집"에 목표 건수가 없음 — "국내 최소 3건·해외 최소 2건 등 총 N건" 식으로 수량 기준을 넣으면 완료 판정이 명확해짐  
### 실용성
- 잘된점: "python generate_newsletter.py ... --send"로 렌더링·발송 실행 명령까지 제공해 즉시 재현 가능함  
- 보완할점: "Gmail로 정상 발송 완료"의 확인 근거(수신 확인·발송 로그)가 미지정임 — "발송 성공 응답/로그 캡처 확인"을 완료조건에 추가하면 정직한 검증이 담보됨  
### 역할별 팀원 배분
- 잘된점: "AI뉴스 수집 전문가"로 역할 정체성은 부여했고 처리 단계를 수집·정제·렌더·발송으로 논리 분해함  
- 보완할점: 단일 역할이 전 과정을 담당해 병렬 수집 이점을 못 살림 — "사이트별 수집 서브에이전트 + 번역 담당 + 편집/발송 담당"으로 역할을 나눠 위임하면 과부하가 해소됨

#### 과제3
## 요약
### 잘된점
- "정적 검증(구문·import 경로·라우터 등록 확인)"으로 완료기준을 측정 가능하게 규정함  
- "실제 서버 기동·응답 확인은 미검증 항목으로 system_arch.md에 명시"로 정직한 보고 원칙을 지킴  
- MUST NOT venv 금지에 "이관 후 재사용 불가"라는 근거를 붙여 제약의 이유를 명확히 함  
### 보완할점
- "백엔드 전문가" 단일 역할로 조사·구현·문서화를 한 주체에 집중시킴  
- "최신 권장 프로젝트 구조 조사"의 판정기준(무엇을 최신으로 볼지)이 열려 있음  
- 라우터·모델의 최소 개수·엔드포인트 범위가 미지정이라 산출 규모가 모호함  
## 설명
### 구체성
- 잘된점: "results/fastapi_app/ 하위에 엔트리포인트, 라우터, 모델, /health 헬스체크 엔드포인트 작성"으로 산출 파일을 구체적으로 특정함  
- 보완할점: "최신 권장 프로젝트 구조·라이브러리 조사"의 채택 기준이 모호함 — "최근 1년 커밋 저장소 3곳 이상 교차 확인, 2곳 이상 공통 채택 관례만 반영" 식으로 판정 규칙을 명시하면 재현 가능함  
### 실용성
- 잘된점: "dev/prod 환경변수 분리 .env.example", "Gunicorn+Uvicorn 워커·Dockerfile"로 상용 이관까지 바로 쓰는 산출물을 규정함  
- 보완할점: requirements.txt의 "버전 명시"만 있고 버전 확정 근거가 미지정임 — "PyPI 최신 안정 버전 기준, 확인 불가 시 x.x.x 플레이스홀더" 규칙을 넣으면 재현성이 높아짐  
### 역할별 팀원 배분
- 잘된점: "Python 3.14 및 FastAPI 개발 경력 10년차 백엔드 전문가"로 역할 전문성을 명확히 정의함  
- 보완할점: 본질상 세팅 단일 흐름이나 "구조 조사·코드 구현·문서화"는 분리 가능함 — 조사 담당과 검증/문서 담당을 나눠 위임하면 정적 검증의 독립 리뷰 효과까지 얻을 수 있음
