## 상세 피드백

#### 과제1
## 요약
### 잘된점
- 완료기준을 "구간별 이동 경로(소요시간·요금)와 맛집 TOP 3(거리·영업시간) 표 제시"로 검증 가능하게 특정함  
- 입력정보(광교중앙역→이태원, 대중교통, 4명)와 사용 도구(GetPublicTransitDirections 등)를 구체적으로 명시함  
- "조회 결과에 없는 수치는 임의 생성 금지" 제약으로 환각 방지·실무 신뢰성을 확보함  
### 보완할점
- 맥락에서 "방문 인원은 좌석 규모 판단 참고"라 했으나 출력 표에 좌석 규모 항목이 없어 입력이 미활용됨(구체성)  
- "거리순 상위 3개" 외 동거리·리뷰수 등 동점 처리 기준이 없어 선정 재현성이 흔들림(실용성)  
- 경로 조회와 맛집 검색이 독립 작업인데 단일 어시스턴트에 순차 집중함(역할배분)  
## 설명
### 구체성
- 잘된점: "구간별 이동 수단·소요시간, 총 소요시간, 요금 / 표(순위·장소명·거리·영업시간)"로 산출물 형태를 측정 가능하게 특정함  
- 보완할점: "방문 인원: 4명"을 참고 정보로만 두고 출력에 반영 안 함. 표에 "수용 좌석/단체 가능 여부" 열을 추가해 입력을 결과에 연결하면 모호함이 사라짐(구체성)  
### 실용성
- 잘된점: "실제 조회(실행 증거) 없이 결과를 완료로 보고하지 않음" 제약이 결과물의 실무 신뢰도를 높임  
- 보완할점: "거리순 상위 3개"만으로는 동거리 시 결과가 달라짐. "거리 동일 시 영업중 우선" 같은 tie-break 규칙을 넣어 재현성을 보강하면 좋음(실용성)  
### 역할별 팀원 배분
- 잘된점: "길찾기·주변 검색 어시스턴트(PlayMCP KakaoMap 도구 활용)"로 작업 성격에 맞는 도구 담당을 지정함  
- 보완할점: 경로 조회·맛집 검색은 상호 독립이라 병렬 가능하나 단일 주체 순차 처리로 둠. 단일 단계에 가까워 과감점은 아니나, 두 하위 조회를 병렬 명시하면 처리 여지가 커짐(역할배분)

#### 과제2
## 요약
### 잘된점
- 6개 지표(S&P500·다우·나스닥·코스피·환율·WTI·VIX·코스피200선물)와 뉴스 5건을 항목 단위로 구체 특정함  
- 데이터 소스 우선순위(MCP 우선→없으면 curl 저장 참조)와 완료조건(발송 로그 확인)을 명확히 규정함  
- WHY("매일 아침 전일 동향 파악 개인 투자자 겸 전문강사")와 재사용 루틴 목적이 분명함  
### 보완할점
- 상단에 빈 [목표]/[역할] 템플릿 블록이 중복 잔존해 실제 지시와 혼선 소지가 있음(구체성)  
- "관련 뉴스 5건" 선정 기준(영향도·최신순 등)이 없어 뉴스 선별이 주관적일 수 있음(실용성)  
- 지표 6종 수집과 뉴스 5건 수집이 병렬 가능하나 단일 역할에 순차 집중함(역할배분)  
## 설명
### 구체성
- 잘된점: "미국 증시: S&P500, 다우존스, 나스닥 … 공포지수: VIX … 코스피200 선물"처럼 지표를 열거해 조회 대상이 모호하지 않음  
- 보완할점: 상단 빈 [목표]~[제약조건] 블록이 그대로 남아 실제 프롬프트와 병존함. 잔여 템플릿을 제거해 지시를 단일화하면 실행 혼선을 줄일 수 있음(구체성)  
### 실용성
- 잘된점: "MCP 없으면 curl {URL} > .temp/{filename}로 저장 후 참조, WebFetch 금지"로 재현 가능한 수집 절차를 규정함  
- 보완할점: "국내 주식에 영향 미칠 뉴스 5건"의 선정 판정기준이 없음. "영향도 높은 순 5건, 발행 24시간 내" 같은 필터를 명시하면 결과 품질이 안정됨(실용성)  
### 역할별 팀원 배분
- 잘된점: "증권 정보 수집 전문가(담당 팀원: 증권 애널리스트/한지민, agent.md 참조)"로 전문성 위임을 명시함  
- 보완할점: 지표 수집·뉴스 수집은 독립 병렬 작업이나 단일 역할에 몰림. 루틴 특성상 단일 주체가 자연스러워 과감점은 아니나, 수집·검증 분리 또는 병렬 지시를 두면 처리량 여지가 늘어남(역할배분)

#### 과제3
## 요약
### 잘된점
- 오케스트레이터+6개 전문 에이전트(큐비·오토·로디·쉴드·유엑스·인사이트)로 QA 전 영역을 전문성별로 배분함  
- 교차검증 게이트(결함 2회 이상 재현, SLO 대비 수치, OWASP 매핑+위험도, 우선순위)로 완료 판정기준을 정량화함  
- "소스 직접 수정 금지·원본 데이터 사본 테스트·비밀값 마스킹"으로 read-only 안전 제약을 명확히 함  
### 보완할점
- "핵심 발견 3~5건" 외 심각도별 최소 커버리지(예: 페이지 3종 전수) 목표치가 없어 테스트 충분성 판단이 애매함(구체성)  
- 로디의 "SLO 대비 수치"에서 기준값(SLO) 자체가 프롬프트에 미제시되어 판정 기준이 외부 의존적임(실용성)  
- 멤버 산출물 형식(공통 템플릿)이 명시 안 돼 인사이트의 통합 단계에서 재작업 소지가 있음(역할배분)  
## 설명
### 구체성
- 잘된점: "페이지 3종(/, /summary, /admin)과 핵심 기능(입력·조회·수정·Excel import/export·AI 요약) 테스트 + 증거 첨부 + 통합 리포트 1건"으로 완료기준을 검증  
  가능하게 특정함  
- 보완할점: 발견 건수 외 커버리지 목표가 없음. "페이지 3종 각 정상·경계·예외 케이스 최소 N건" 같은 수행 하한을 넣으면 테스트 충분성을 객관화할 수 있음(구체성)  
### 실용성
- 잘된점: "AI 요약 프롬프트 인젝션(Excel 입력 텍스트를 통한 LLM 지시 탈취·민감정보 유출) 점검"까지 실제 위험을 짚어 실무 즉시 활용도가 높음  
- 보완할점: "기준값(SLO) 대비 수치 명시"라 했으나 SLO 목표치(응답시간 몇 ms 등)가 프롬프트에 없음. 기본 SLO를 예시로 제시하면 성능 판정이 재현 가능해짐(실용성)  
### 역할별 팀원 배분
- 잘된점: "인사이트: 멤버 산출물을 심각도·영향도·재현성으로 분류·통합"으로 계층형+협업형 패턴을 작업 구조에 맞게 설계해 단일 주체 과부하를 제거함  
- 보완할점: 각 멤버 산출물의 공통 항목·형식이 미정이라 통합 시 편차가 생길 수 있음. "멤버별 발견은 발견사항 표 형식으로 통일 제출"을 지시하면 인사이트 통합이 매끄러워짐(역할배분)
