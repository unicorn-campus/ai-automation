# 2회차 과제 평가 피드백 — 고광진

> 평가 관점: 구체성·실용성·역할별 팀원 배분 (각 100점, 동일 가중). 과제별 점수는 3개 관점 평균, 종합점수는 3개 과제 평균임. 복제 확정 과제는 0점 처리.

## 상세 피드백

#### 과제1
## 요약
### 잘된점
- 산출물·완료조건·타임스탬프 형식(`YYYYMMDD-HHmmss`)까지 특정해 검증 가능함  
- WHY("메뉴 고민 시간을 줄임")가 명확하고 입력 플레이스홀더로 재사용 가능함  
- 미조회 항목을 "미확인"으로 강제해 환각 방지 장치가 실무적임  
### 보완할점
- "점심 메이트" 단일 역할에 조회·선별·경로 확인이 몰려 있음(역할배분)  
- "강남역 도보권" 판정이 주소 예시 위주라 임계 거리 기준이 모호함(구체성)  
- 후보 식당 개수 "1~2곳"의 선정 우선순위 기준이 미정의됨(실용성)  
## 설명
### 구체성
- 잘된점: "`outputs/{timestamp}.md` 파일이 생성됨"과 "`## 오늘의 추천`과 `## 메뉴별 후보`가 모두 채워진"으로 완료조건을 측정 가능하게 명시함  
- 보완할점: "강남역 도보권(강남대로·테헤란로 초입)"이 예시 지명 나열에 그침. "3번출구 도보 15분/1km 이내"처럼 임계값을 수치로 못박으면 판정이 재현 가능해짐  
### 실용성
- 잘된점: "검색 키워드는 '강남역 {메뉴}' 형식"과 "출구까지 넣으면 검색이 잘 안 됨 — 시행착오 반영"으로 실행 노하우를 담아 바로 재현 가능함  
- 보완할점: "메뉴별 후보 식당 1~2곳을 정리"에서 2곳 중 무엇을 남길지 기준이 없음. "리뷰수·거리 우선"처럼 선정 규칙을 넣으면 결과 편차가 줄어듦  
### 역할별 팀원 배분
- 잘된점: "당신은 강남역 주변 상권에 밝은 점심 메이트입니다"로 도메인 페르소나를 명확히 부여함  
- 보완할점: 조회·필터·경로확인을 단일 역할이 수행함. 본질상 단일 단계에 가까워 과감점은 않으나, 메뉴별 검색을 병렬 태스크로 분해하면 처리 속도·재현성이 개선됨

#### 과제2
## 요약
### 잘된점
- UI/분류/데이터/렌더러를 분리하고 JSON 스키마·파일명 규칙까지 특정해 매우 구체적임  
- "메일 발송 성공 로그 확인"을 완료조건에 넣어 정직한 보고 원칙에 부합함  
- 우선순위(펌웨어·부트로더 취약점 우선)로 개발자 관점 가치(WHY)가 뚜렷함  
### 보완할점
- 수집→정제→렌더링→발송 전 단계를 단일 큐레이터에 몰아 병렬화 여지를 못 살림(역할배분)  
- 국내외 8개 출처 병렬 수집을 서브에이전트로 분해하지 않음(역할배분)  
- "지난 24시간" 판정의 기준 시각(수집시점 vs KST 00시)이 모호함(구체성)  
## 설명
### 구체성
- 잘된점: `articles` 항목의 category/title/url/summary/source/time 필드와 "요약은 200자 내"로 데이터 규격을 검증 가능하게 못박음  
- 보완할점: "최근 24시간 뉴스만 필터링"의 기준점이 불명확함. "`published_at` 기준 -24h 이내"처럼 앵커 시각을 명시하면 경계 판정이 재현 가능해짐  
### 실용성
- 잘된점: "`python generate_newsletter.py data/... --send`"로 실행 명령까지 제시해 루틴으로 즉시 반복 실행 가능함  
- 보완할점: 출처 8곳이 모두 접근 실패할 때의 대체 동작이 없음. "출처 3곳 미만 수집 시 발송 보류·알림"류 예외 규칙을 넣으면 실무 안정성이 올라감  
### 역할별 팀원 배분
- 잘된점: "테크 뉴스 큐레이터임"으로 선별·요약 전문성을 명시함  
- 보완할점: "위 출처를 웹 검색으로 지난 24시간 내 기사만 병렬 수집"이라 하나 실행 주체가 단일임. 수집(리서처)·요약번역(작성자)·검수(리뷰어) 파이프라인으로 분해하면 과부하 해소·품질 검증이 가능함

#### 과제3
## 요약
### 잘된점
- 분류·출력 순서·해시 표기·"(내용 확인 필요)"까지 규정해 산출물이 검증 가능함  
- "분류별 건수 표와 본문 항목 수가 일치함"으로 정합성 완료조건을 수치화함  
- 커밋→사용자 관점 변환 예시가 구체적이라 WHY와 활용성이 분명함  
### 보완할점
- "릴리즈 매니저" 단일 역할이 분류·변환·검수를 모두 수행함(역할배분)  
- 보안 패치의 "영향 범위" 표기 수위 기준이 정성적이라 모호함(구체성)  
- 대량 커밋 시 항목 누락 검증 절차가 없음(실용성)  
## 설명
### 구체성
- 잘된점: "각 항목 끝에 근거 커밋 해시를 괄호로 표기 (예: `(a1f3c92)`)"와 분류 5종을 명시해 출력 형식이 검증 가능함  
- 보완할점: "보안 패치는 공격 방법·재현 절차를 쓰지 않고 영향 범위와 조치 결과만 기술"에서 '영향 범위'의 서술 수위가 불명확함. "영향받는 기능·버전만, 취약점 유형명 제외"처럼 허용/금지 항목을  
  열거하면 판정이 명확해짐  
### 실용성
- 잘된점: "릴리즈 때마다 커밋 로그를 수작업으로 분류·정리"라는 반복 고통점을 정조준해 실무 재사용성이 높음  
- 보완할점: "커밋 메시지만으로 의미가 불명확한 항목은 (내용 확인 필요) 표시"는 좋으나, 전체 커밋 수 대비 처리 건수 대조가 없음. "입력 커밋 N건 = 본문+내부변경+미확인 합계" 검증을 완료조건에  
  추가하면 누락 방지됨  
### 역할별 팀원 배분
- 잘된점: "개발자의 커밋 메시지를 QA·기술지원·고객 관점의 설명으로 바꾸는 일에 익숙합니다"로 독자 지향 전문성을 명확히 부여함  
- 보완할점: 본질상 단일 변환 작업이라 과감점은 않으나, 초안 작성(릴리즈 매니저)과 사용자 관점 가독성 검수(기술지원 리뷰어)를 분리하면 고객사 FAE 대상 품질이 개선됨
