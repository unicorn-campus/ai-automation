## 상세 피드백

#### 과제1
## 요약
### 잘된점
- 출력 예시(이동 경로 단계·요금, "주변 맛집 TOP 3" 표)를 제시해 결과 형태가 눈에 보임  
- 입력 항목(출발지·목적지·이동 수단, 주변 검색·방문 인원)을 명확히 나열함  
- 카카오맵 MCP라는 실제 실행 도구를 특정해 바로 호출 가능함  
### 보완할점
- 맛집 선정 판정기준(거리순·평점순 등)이 없어 TOP 3 기준이 모호함 (구체성)  
- WHY(왜 필요한지)와 완료조건이 없어 실무 재현성이 약함 (실용성)  
- [역할] 섹션이 없어 전문가 지정·위임이 부재함 (역할배분)  
## 설명
### 구체성
- 잘된점: "주변 맛집 TOP 3 (블루스퀘어 반경 500m)" 표와 "요금: 1,400원" 등 출력 형식을 구체 예시로 고정함  
- 보완할점: "'주변 검색'에 맞는 키워드로 목적지의 주변을 검색한다"만 있고 TOP 3 정렬 기준이 없음.  
  "반경 500m 내 카카오맵 평점순 상위 3곳" 처럼 선정·정렬 판정기준을 명시하면 결과가 재현됨  
### 실용성
- 잘된점: "카카오맵 MCP를 활용해 이동 경로를 출력한다"로 실행 도구가 분명해 즉시 실행 가능함  
- 보완할점: WHY와 완료조건이 없음. "이동 경로+주변 맛집을 한 번에 확보해 방문 계획 시간을 줄임" 같은  
  기대가치와 "경로·맛집 표 출력 완료"를 완료조건으로 넣으면 실무 활용도가 올라감. 오타("겸색")도 정정 필요  
### 역할별 팀원 배분
- 잘된점: 단일 단계(경로 조회+주변 검색) 작업으로 성격상 단일 주체 수행이 무리 없음  
- 보완할점: "당신은 지도·장소 검색 전문가임"처럼 [역할]을 명시하지 않아 전문성 근거가 없음.  
  경로 조회와 맛집 검색을 병렬 호출하도록 역할·에이전트 위임을 덧붙이면 재사용성이 개선됨

#### 과제2
## 요약
### 잘된점
- 산출물 경로(data/·newsletters/·categories.json)와 JSON 스키마를 구체적으로 명시함  
- 완료조건(데이터·HTML 생성, Slack·Gmail 발송, git push)을 검증 가능하게 특정함  
- "에이전트 호출하여 병렬로 수집"으로 수집 단계의 병렬화를 살림 (역할배분)  
### 보완할점
- [목표]는 "경제 뉴스"인데 [입력] 라벨은 "AI 뉴스 사이트", 예시 sources도 "AI Times/지디넷"로 불일치 (구체성)  
- 수집 에이전트만 병렬이고 정제·번역·렌더링 주체가 명시되지 않음 (역할배분)  
- "최근 24시간" 판정 기준시각(발행시각/수집시각)이 모호함 (구체성)  
## 설명
### 구체성
- 잘된점: "data/YYYYMMDD_HHMM.json 으로 저장" + JSON 스키마와 "summary: 200자 내 핵심 요약"로 형식을 정량화함  
- 보완할점: [입력] 제목이 "AI 뉴스 사이트"이고 sources 예시가 "AI Times, 지디넷"으로 남아 목표(경제)와 어긋남.  
  라벨을 "경제 뉴스 사이트", 예시를 매경/한경 등으로 통일하면 지시 혼선이 사라짐  
### 실용성
- 잘된점: "내상황: 새로운 경제 정보를 얻고 주식시장에 활용하고 싶은 일반인임"으로 WHY가 드러남  
- 보완할점: UI·분류·데이터·렌더러 분리 구조로 재현성은 높으나, "24시간" 기준시각이 불명확함.  
  "각 기사 발행시각(published time) 기준 24시간 이내"로 못박으면 필터 결과가 재현됨  
### 역할별 팀원 배분
- 잘된점: "경제 뉴스 사이트에서 ... 에이전트 호출하여 병렬로 수집"으로 수집 병목을 병렬로 분해함  
- 보완할점: 수집 외 정제·번역·분류를 누가 맡는지 미지정. "번역 에이전트/분류 에이전트"로 파이프라인을  
  단계별 위임하면 단일 주체 과부하를 줄이고 협업 패턴이 명확해짐

#### 과제3
## 요약
### 잘된점
- 요청 유형별(A 구현/B 장애/C 성능/D 리뷰) 처리 절차를 분기해 검증 가능하게 구조화함  
- MUST NOT "실행·검증하지 않은 것을 동작 확인 완료로 보고 금지"로 판정기준을 엄격히 못박음  
- 긴 입력을 XML 태그(<log>·<pcap> 등)로 감싸도록 지정해 지시·데이터 혼용을 방지함 (구체성)  
### 보완할점
- 작업 환경 값이 "<예: C/C++, Python>"처럼 플레이스홀더로 남아 실제 특정이 미완임 (구체성)  
- 단일 전문가에 A~D 전 업무를 집중시켜 병렬·분해 여지를 살리지 못함 (역할배분)  
- 성능 "기준선 확보"의 목표 수치·수용 임계값이 정량화되지 않음 (구체성)  
## 설명
### 구체성
- 잘된점: 출력을 "증상 → 원인 → 근거 → 조치 → 잔여 리스크 순서로 구조화"로 고정하고 완료조건을 증거 확인으로 명시함  
- 보완할점: [맥락] 작업 환경이 "<예: ...>" 예시로만 있어 대상이 특정되지 않음.  
  실제 언어·프로토콜·대상 스택을 요청 시 반드시 채우도록 필수 입력으로 승격하면 지시가 명확해짐  
### 실용성
- 잘된점: "근거 우선: 추정이 아니라 표준 문서(RFC)·실제 로그·패킷 캡처·측정 결과로 판단함"으로 실무 신뢰성을 확보함  
- 보완할점: [예시]의 gRPC 타임아웃 케이스는 재사용성이 높으나 성능 수용 기준이 없음.  
  "p99 지연 X ms 이하" 같은 수용 임계값을 예시에 넣으면 개선 효과 판정이 곧바로 가능해짐  
### 역할별 팀원 배분
- 잘된점: 본질적으로 단일 개발자 역할 프롬프트로, "네트워크 소프트웨어 개발 전문가" 전문성 정의가 명확함  
- 보완할점: A~D 유형을 한 주체가 모두 수행함. 특히 (D) 코드/설정 리뷰는 별도 리뷰어에게 위임하면  
  구현자-리뷰어 교차검증(협업형)이 되어 품질과 병렬성이 함께 개선됨
