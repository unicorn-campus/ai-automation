## 상세 피드백

#### 과제1
## 요약
### 잘된점
- 산출물·완료조건이 `outputs/route-food-result.md`와 "이동 경로"·"주변 맛집 TOP 3" 표로 명확히 특정됨  
- 입력 형식·예시(출발지/목적지/이동수단/주변검색/방문인원)를 구체 제시해 재현 가능함  
- MUST NOT로 "조회하지 않은 경로·맛집 정보를 지어내지 않음" 명시해 환각 방지 장치가 견고함  
### 보완할점
- "평점·거리·영업시간을 기준으로 상위 후보"의 정렬·가중 기준이 미명시라 판정기준이 모호함(구체성)  
- 프롬프트 본문이 단일 전문가 관점이라 AGENTS.md의 파이프라인 위임이 실행에 반영될지 불명확함(역할배분)  
- "방문 인원 수용 가능 매장 우선"의 수용 여부 확인 방법(테이블수/단체석)이 없어 실무 적용이 약함(실용성)  
## 설명
### 구체성
- 잘된점: "route-food-result.md 파일이 실제 생성됨", "표가 채워진 것을 확인함" 등 완료조건을 동작 확인 기준으로 특정함  
- 보완할점: "평점·거리·영업시간을 기준으로 상위 후보를 정리함"은 기준 간 우선순위가 없어 "거리 우선 후 평점 4.0 이상, 동점 시 도보 5분 내" 식으로 정렬규칙을 수치화하면 재현성이 오름  
### 실용성
- 잘된점: "처음 방문하는 지역에서 이동경로와 맛집을 한눈에 확인"으로 WHY와 독자가 분명해 실무 가치가 명확함  
- 보완할점: "방문 인원 수용 가능 매장을 우선"만 있고 6명 수용 확인 근거가 없어, MCP 검색 결과에 단체석·예약 필드가 없으면 "확인 불가로 표기" 규칙을 추가해야 실제 이용 실패를 막음  
### 역할별 팀원 배분
- 잘된점: AGENTS.md에서 "경로 관련은 길찾기 담당, 맛집 관련은 맛집 탐색가에게 위임" 후 정리 작성자가 통합하는 파이프라인이 작업 구조(경로→맛집→정리)에 적합함  
- 보완할점: 프롬프트 [역할]이 "길찾기 맛집추천 전문가" 단일 주체라 위임 흐름이 본문에 드러나지 않음. 과제2처럼 "경로·맛집 조회를 병렬 에이전트로 호출" 문구를 [처리]에 넣으면 분해·병렬 여지가 실행에  
  반영됨

#### 과제2
## 요약
### 잘된점
- `data/YYYYMMDD_HHMM.json` 스키마를 필드 단위로 제시하고 절대경로 산출물 3종을 특정해 재현성이 높음  
- "뉴스 수집: ...에이전트 호출하여 병렬로 수집"으로 병렬화를 본문에 명시해 분해 의도가 실행에 드러남  
- 완료조건을 JSON·HTML 생성, Slack·메일 발송, git push 확인으로 나눠 동작 검증 기준이 촘촘함  
### 보완할점
- "최근 24시간"의 기준 시각(published_at KST 대비 수집 시점)이 경계에서 애매해 판정 오차 여지가 있음(구체성)  
- Slack·메일·git push는 부작용 동반 작업인데 발송 실패 시 재시도·롤백 규칙이 없어 운영 견고성이 약함(실용성)  
- 발송이가 렌더·메일·Slack·push를 모두 담당해 단일 주체 과부하 여지가 있음(역할배분)## 설명  
### 구체성
- 잘된점: JSON 예시에 `category`/`category_label`/`category_emoji` 신규 카테고리 규칙까지 "함께 기입(필수)"로 명문화해 모호함이 거의 없음  
- 보완할점: "'최근 24시간' 이전에 올라온 뉴스 포함 금지"는 있으나 기준 시각 정의가 없어 "수집 실행 시각 기준 -24h, 게시시각 파싱 불가 시 제외" 식으로 경계 판정을 명시하면 오분류가 줄어듦  
### 실용성
- 잘된점: "매일 아침 8시" 정기 루틴과 렌더러·발송·push까지 완결 워크플로우라 실무 자동화로 바로 활용 가능함  
- 보완할점: "Slack과 메일로 정상 발송 완료 확인" 완료조건만 있고 실패 대응이 없어, "메일 실패 시 재시도 1회 후 오케스트레이터 보고" 같은 예외 처리 규칙을 [처리]에 추가하면 운영 안정성이 오름  
### 역할별 팀원 배분
- 잘된점: AGENTS.md에서 수집이(병렬수집·중복제거)→편집이(요약·번역·분류·JSON)→발송이(렌더·발송·push)로 파이프라인이 작업 순서와 정확히 일치함  
- 보완할점: 발송 담당이 렌더·메일·Slack·push를 단독 수행해 병목 우려가 있음. 발송(메일/Slack)과 배포(git push)를 분리하거나 발송 채널을 병렬 처리하도록 위임 규칙을 세분화하면 과부하가  
  줄어듦

#### 과제3
## 요약
### 잘된점
- 안정성·보안·베스트프랙티스 점검 항목을 requests/limits·probe·privileged·latest 태그 등으로 열거해 검증 가능함  
- 출력 표 열(위치/심각도/유형/설명/조치안)과 예시 1행을 제시해 산출물 형태가 명확함  
- "불확실은 추정으로 표기", "실제 시크릿 값 노출 금지"로 정확성·보안 가드레일이 견고함  
### 보완할점
- 심각도 상/중/하를 "장애·보안 영향도 기준"으로만 부여해 등급 경계가 주관적일 여지가 있음(구체성)  
- 다양한 입력 형식(YAML/Dockerfile/Terraform/CI) 대비 형식별 점검 깊이 차이가 명시되지 않음(실용성)  
- 프롬프트 [처리] 1~3단계가 전문 영역별로 나뉘나 본문은 단일 리뷰어 관점이라 위임이 드러나지 않음(역할배분)## 설명  
### 구체성
- 잘된점: "리소스 requests/limits 누락, liveness/readiness probe 유무, replica/HPA 설정, PDB 존재 여부"처럼 점검 대상을 구체 항목으로 특정함  
- 보완할점: "심각도는 장애·보안 영향도 기준으로 상/중/하로 일관 부여"는 기준이 서술적임. "상=서비스 중단/시크릿 노출, 중=성능·가용성 저하, 하=권고사항" 식 등급 정의를 넣으면 일관성이 검증 가능해짐  
### 실용성
- 잘된점: "K8s manifest·Dockerfile·Terraform/Helm·CI yml, 경로 또는 텍스트, 형식 무관"으로 입력 유연성이 높아 실무 재사용에 유리함  
- 보완할점: 형식 무관 입력을 받되 형식별 점검 항목 매핑이 없어, "Dockerfile은 태그 핀·비루트 유저, Terraform은 상태·권한" 식으로 파일유형별 체크리스트를 분기하면 누락 없이 적용됨  
### 역할별 팀원 배분
- 잘된점: AGENTS.md에서 안정성(민호)·보안(서연)·정리(지훈)로 분리하고 "항목이 겹치면 승재가 담당 단일화·중복 병합"까지 규정해 병렬·충돌해소 설계가 우수함  
- 보완할점: 프롬프트 [처리] 1(안정성)·2(보안)은 병렬 가능한데 본문이 순차 단일 주체로 서술됨. "안정성·보안 점검을 병렬 에이전트로 수행 후 작성자가 통합" 문구를 넣으면 AGENTS.md 병렬 설계가  
  실행에 반영됨
