## 상세 피드백

#### 과제1
## 요약
### 잘된점
- 도구명(KakaoMap-GetPublicTransitDirections/SearchPlaceByKeywordOpen)·반경 500m·TOP 3를 특정하여 지시가 검증 가능함  
- WHY(처음 가는 목적지 대중교통+맛집 한 번에 파악)가 맥락에 분명히 서술되어 실무 가치가 뚜렷함  
- 입출력 예시와 완료조건 2개를 제시하여 재현·판정 기준을 갖춤

### 보완할점
- 맛집 선별 판정기준이 "거리·영업 여부 종합"뿐으로 가중치·우선순위가 모호함(구체성)  
- 작업방법에서 "평점을 수집"하나 출력 표(순위·장소명·거리·영업시간)엔 미반영, 인원 반영도 예시서 누락(실용성)  
- AGENTS.md에 경로탐색가·맛집큐레이터·데이터검증가 3인이 있으나 프롬프트가 단일 어시스턴트로 몰아 위임 부재(역할배분)  
## 설명
### 구체성
- 잘된점: "목적지 반경 500m 내 검색 키워드에 맞는 장소를 조회", "상위 3곳을 TOP 3로 선별"로 산출물이 측정 가능하게 특정됨  
- 보완할점: "목적지 기준 거리·영업 여부를 종합하여 상위 3곳"은 종합 기준이 모호함. 거리 임계·영업중 필수 여부·평점 가중치를 순위 산식으로 명시하면 판정이 재현 가능해짐  
### 실용성
- 잘된점: "카카오맵 MCP 조회 없이 임의 추정으로 경로·맛집 정보 생성 금지"로 근거 기반 실무 산출을 강제함  
- 보완할점: "방문 인원 수를 고려하여 단체 수용 가능 여부를 추천 사유에 반영"·"평점을 수집"이 예시 출력엔 빠짐. 출력 표에 평점·단체수용 열을 추가해 지시-산출을 일치시키면 실무 활용도가 오름  
### 역할별 팀원 배분
- 잘된점: 작업방법을 경로 조회(1)→맛집 조회(2)→선별(3)→교차 검증(4) 단계로 분해하여 검증가 역할(4단계 최신성 교차검증)을 절차에 반영함  
- 보완할점: AGENTS.md의 "이동경로 탐색→경로탐색가, 맛집 큐레이션→맛집큐레이터, 팩트체크→데이터검증가" 매핑이 있음에도 커넥터 프롬프트가 위임을 전혀 쓰지 않음. 경로·맛집을 병렬 위임 후 검증가가  
  교차확인하는 구조로 바꾸면 과부하·병렬화 여지를 살릴 수 있음

#### 과제2
## 요약
### 잘된점
- UI/분류/데이터/렌더러/출력을 파일 경로로 분리 정의하여 작업 경계가 명확하고 재현 가능함  
- data JSON 스키마 예시와 "200자 내 요약"·"최근 24시간 필터"로 산출·완료 기준을 구체화함  
- 완료조건 3개(JSON/HTML 생성, Slack·Gmail 발송, git push)로 판정 기준이 검증 가능함

### 보완할점
- [역할]이 "등산 뉴스 수집 전문가"인데 실제 작업은 수집+번역+렌더링+발송+push까지로 역할-범위 불일치(역할배분)  
- Slack 발송이 "'#ai-news' 채널로 발송"으로 채널명이 등산 주제와 어긋나고 발송 방법이 메일보다 덜 구체적임(구체성)  
- "에이전트 호출하여 병렬로 수집"만 언급하고 에이전트 정의·분담이 없어 병렬화가 선언에 그침(실용성/역할배분)  
## 설명
### 구체성
- 잘된점: "`data/YYYYMMDD_HHMM.json`으로 저장" 및 JSON 필드(published_at·source·category)까지 명시해 출력이 모호하지 않음  
- 보완할점: Slack은 "'#ai-news' 채널로 발송"만 있어 메일(`--send` 명령)과 구체성 격차가 큼. 등산 주제에 맞는 채널명·발송 명령/도구를 명시하면 실행 편차가 줄어듬  
### 실용성
- 잘된점: "python generate_newsletter.py data/....json --send"처럼 실행 명령을 그대로 제시해 루틴 자동화·재현이 가능함  
- 보완할점: "등산 뉴스 사이트에서 지난 24시간 내에 올라온 뉴스를 에이전트 호출하여 병렬로 수집"이 에이전트 실체 없이 선언됨. 사이트군을 나눠 병렬 수집 단위를 구체화하면 실제 실행 가능성이 높아짐  
### 역할별 팀원 배분
- 잘된점: "에이전트 호출하여 병렬로 수집"으로 수집 단계의 병렬화 의도를 명시하여 단일 주체 과부하를 인식함  
- 보완할점: [역할]="등산 뉴스 수집 전문가" 단일 규정과 수집·번역·발송을 한 주체에 몰아 배분이 사실상 없음. 수집/번역·정제/발송·검증 역할을 나눠 병렬 위임 구조로 정의하면 파이프라인이 견고해짐

#### 과제3
## 요약
### 잘된점
- 10개 추정기·참조구현 ea_iid/ea_non_iid/ea_restart·샘플 수 ≥1,000,000까지 특정해 지시가 극도로 검증 가능함  
- "데이터 실행 없이 기억·추론만으로 min-entropy 값을 작성하지 말 것" 등으로 재현·정직 기준이 명확함  
- 단일 평가 프롬프트와 팀 오케스트레이션 프롬프트를 분리하여 작업 성격별 구조가 최적화됨

### 보완할점
- 채택 최솟값 산식은 있으나 "핵심 평가 항목 1~3개 파악" 상한 근거가 다소 임의적임(구체성)  
- 예시 출력의 추정값·일치 표가 가상 수치라 실제 실행값과 혼동 여지가 있어 예시임을 더 분명히 하면 좋음(실용성)  
- 컨디셔닝 지원이 프롬프트B에서 "요청 범위 외 미구현"으로 남아 후속 편입 기준을 명시하면 완결성이 오름(역할배분)  
## 설명
### 구체성
- 잘된점: "non-IID track: MCV·Collision·Markov·Compression·t-Tuple·LRS·MultiMCW·Lag·MultiMMC·LZ78Y 10개 추정기 실행 후 최솟값을  
  min-entropy로 채택"으로 판정기준이 계산 가능하게 특정됨  
- 보완할점: "핵심 평가 항목 1~3개 파악"의 개수 상한 근거가 약함. track·재시작 유무 등 입력 신호별로 항목이 결정되는 규칙을 붙이면 판정이 더 일관됨  
### 실용성
- 잘된점: "원시 데이터를 코드 실행으로 1회 이상 실제 로드·추정한 결과를 인용"으로 실무 재현·근거 인용을 강제함  
- 보완할점: 예시 출력의 추정값 표(6.82/5.91 등)가 실제 실행값과 구분이 약함. 예시 표에 "(예시 값)" 라벨을 명시하면 실행 결과와의 혼동을 방지함  
### 역할별 팀원 배분
- 잘된점: 프롬프트B가 "파이프라인형 뼈대(표준분석 → 구현 → 검증 → 보고), 단계 내 설계 판단은 토론형"으로 AGENTS.md 5인(스탠다·트로피·큐에이·리포·클로니)에 정확히 위임함  
- 보완할점: "협업형이 필요한 구간은 클로니가 라운드를 중개함"까지 규정해 완성도가 높음. 다만 "컨디셔닝 지원은 요청 범위 외로 미구현"의 후속 편입 트리거(입력 명시 시 어느 단계 삽입)를 명문화하면 팀  
  확장 시 배분이 더 견고해짐
