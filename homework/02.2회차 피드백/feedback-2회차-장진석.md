## 상세 피드백

#### 과제1
## 요약
### 잘된점
- 오케스트레이터 역할로 "경로 탐색 담당, 맛집 검색 담당, 결과 정리 담당" 3개 하위 역할에 명시 위임함  
- 반경 500m·TOP 3·요금(원)·거리(m)·시간(분) 등 측정 단위와 출력 마크다운 구조를 특정함  
- "MCP로 확인되지 않는 값은 확인 불가로 명시하고 추측하지 않습니다"로 재현성·정직성 확보함  
### 보완할점
- 위임 역할이 실제 조율 방식(입력→출력 전달 규약) 없이 이름만 나열되어 협업 흐름 구체화 필요함  
- "가능하면 단체 수용이 가능한 곳을 우선"의 판정 근거가 모호해 좌석수·룸 유무 기준 명시 필요함  
- MCP 미연결 시 "그 사실을 먼저 알립니다"만 있어 대체 동작·중단 여부 완료조건 보강 필요함  
## 설명
### 구체성
- 잘된점: "맛집 검색 반경 기본 500m(요청 시 조정), 결과는 TOP 3", "요금은 원, 거리는 m, 시간은 분 단위로 표기"로 산출물·단위를 특정함  
- 보완할점: "방문 인원을 고려해 가능하면 단체 수용이 가능한 곳을 우선(불명확하면 표시 생략)"은 판정기준이 모호함. "6명 이상 시 좌석 8석↑ 또는 단체룸 보유 장소 우선"처럼 측정 가능 기준으로 고치면  
  좋음  
### 실용성
- 잘된점: "반드시 실제 MCP 호출 결과를 사용하고, 임의로 지어내지 않습니다"로 재현·실무활용 가치를 명확히 함  
- 보완할점: MCP 미연결 시 "연결이 없으면 그 사실을 먼저 알립니다"만 있어 실행 중단·재시도 여부가 불명확함. 미연결 시 처리 분기(중단/사용자 재연결 안내)를 완료조건에 추가 권장함  
### 역할별 팀원 배분
- 잘된점: "AGENTS.md에 정의된 팀(경로 탐색 담당, 맛집 검색 담당, 결과 정리 담당)에게 작업을 위임하고 결과를 통합"으로 병렬 분해 여지를 살림  
- 보완할점: 각 담당의 입력·산출 규약과 통합 순서가 없어 이름 나열에 그침. "경로 담당 산출(좌표)→맛집 담당 입력"처럼 데이터 전달 규약을 명시하면 조율이 검증 가능해짐

#### 과제2
## 요약
### 잘된점
- 데이터 JSON 스키마·published_at·source 규칙을 예시 코드로 특정해 구체성이 매우 높음  
- MUST/MUST NOT/완료조건을 분리하고 "지정한 메일로 정상 발송 완료" 등 실행 증거 기준을 세움  
- UI·분류·데이터·렌더러 분리 구조로 재사용·재현 가능한 실무 파이프라인을 설계함  
### 보완할점
- [역할]이 "정보보안 뉴스 수집 전문가" 단일이라 정제·렌더·발송 단계 위임 주체가 불명확함  
- "(팀 에이전트로) 병렬 수집" 한 줄 외 팀 분해가 없어 수집 이후 단계가 단일주체에 집중됨  
- "200자 내 핵심 요약"의 품질 판정 기준이 없어 요약 검증 방법 보강 필요함  
## 설명
### 구체성
- 잘된점: articles 필드에 "category, title, url, summary, source, time" JSON 예시와 "source: URL이 아니라 사이트명"까지 특정해 모호함이 거의 없음  
- 보완할점: "내용 요약은 사용자 친화적으로 쉽게, 200자 내로 작성"에서 "쉽게"가 모호어임. "전문용어는 괄호로 1회 풀이, 문장 3개 이내"처럼 검증 가능한 규칙으로 보강하면 좋음  
### 실용성
- 잘된점: "최신 보안 위협·취약점·침해사고 동향을 매일 아침 빠르게 파악하고 싶은 실무자"로 WHY가 명확하고, 렌더러·발송 명령까지 있어 실행·재현 가능함  
- 보완할점: 완료조건 "지정한 메일로 정상 발송 완료"의 증거(발송 로그·수신 확인) 확인 방법이 없어, 발송 성공 판정 근거를 명시하면 정직 보고가 강화됨  
### 역할별 팀원 배분
- 잘된점: "위 사이트에서 지난 24시간 내에 올라온 보안 뉴스를 (팀 에이전트로) 병렬 수집"으로 수집 단계 병렬화 여지를 살림  
- 보완할점: [역할]이 "수집 전문가" 단일이라 정제·번역·렌더·발송 단계 위임이 없어 단일주체 과부하 위험이 있음. 번역 담당·렌더 담당 등 단계별 역할 분해를 추가하면 협업 적합성이 높아짐

#### 과제3
## 요약
### 잘된점
- success=false 신규 샘플만 추출·상태파일 중복방지 등 산출물과 완료동작을 정밀하게 특정함  
- "--dry-run으로 먼저 감지 결과를 검증"으로 실제 발송 전 검증 절차를 제약에 명문화함  
- 콘솔 요약·Slack 메시지 필드·예시 출력까지 제시해 재현 가능한 실무 알림 도구를 설계함  
### 보완할점
- [역할]이 "담당자" 단일 주체라 파싱·감지·발송 단계의 역할 분해·병렬화 여지가 미반영됨  
- "대량이면 상위 N건 + 외 M건"의 N 기준값이 없어 임계 수치 명시 필요함  
- 형식 예외 시 "원인을 보고한다"만 있어 보고 형식·수신자 구체화 필요함  
## 설명
### 구체성
- 잘된점: "success=false 신규 샘플 추출 → 상태파일(.jtl_state.json)로 중복 방지"와 실패 필드 "API명·응답코드·URL·소요(ms)·assertion 실패메시지"로 산출물이 특정됨  
- 보완할점: "대량이면 상위 N건 + 외 M건"에서 N이 미지정임. "상위 10건 초과 시 요약"처럼 임계값을 특정하면 동작이 결정적이 됨  
### 실용성
- 잘된점: "이미 알린 실패는 다시 보내지 않는다", "토큰을 로그·메시지·코드에 노출 금지"로 실무 반복 실행 안전성과 재현성을 확보함  
- 보완할점: "결과 파일이 없거나 형식이 예상과 다르면 발송하지 말고 원인을 보고한다"의 보고 대상·형식이 없어, 콘솔/Slack 중 어디로 보고할지 명시하면 실무 활용도가 높아짐  
### 역할별 팀원 배분
- 잘된점: 단일 도구(jtl_slack_notify.py) 실행이 본질인 태스크로, "담당자"가 파싱→감지→발송을 일관 수행하는 구성이 과제 성격에 부합함  
- 보완할점: 파싱·중복감지·발송이 순차 단일주체에 집중됨. 대량 결과 확장을 대비해 "파싱/감지 담당"과 "발송 담당" 분리 등 분해 여지를 역할에 언급하면 확장성이 커짐
