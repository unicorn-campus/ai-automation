## 상세 피드백

#### 과제1
## 요약
### 잘된점
- 입력 5종·기본값("대중교통/맛집/2명")·"반경 500m" 등 판정기준이 수치로 특정되어 검증 가능함  
- "지도 앱을 일일이 뒤지지 않고 안내문 한 장으로 이동·식사를 결정"으로 WHY가 선명하고 실무 재현 가능함  
- "MCP가 반환하지 않은 상호·주소·영업시간을 지어내지 않음"으로 환각 방지 가드가 명확함  
### 보완할점
- "단체 착석이 가능한 곳을 우선(실내·좌석 여유)"은 MCP가 좌석정보를 반환 못하면 검증 불가 — 대체기준 명시 필요함  
- 페르소나 "카비"가 레포 로스터의 MCP 전담 "커니"와 무관 — 커니 전문성 반영 또는 근거 필요함  
- 작성·검증 분리(베라)가 없어 완료조건 자가확인에 그침 — 검증 패스 1단계 추가 권장함## 설명  
### 구체성
- 잘된점: "거리 가까운 순 → 상위 3곳 선별, 동점 시 영업시간이 넓은 곳 우선", "results/{YYYY-MM-DD}-{목적지}.md" 등 정렬·경로 기준이 구체적임  
- 보완할점: "단체 착석이 가능한 곳을 우선(실내·좌석 여유)"은 MCP 반환값에 좌석 여부가 없으면 판정 불가 — "좌석정보 미제공 시 이 기준 생략" 같은 폴백을 명기하면 완결됨  
### 실용성
- 잘된점: "외출·약속 직전, 출발지/목적지가 정해질 때마다(온디맨드)"로 사용 시점이 실무 밀착적이고 재사용 가능함  
- 보완할점: "MCP 조회 실패 시 실패 사유를 안내문 상단에 명시"는 좋으나, 부분 실패(경로 성공·맛집 실패)의 표 처리 규칙이 없음 — 섹션별 실패 표기 규칙 추가 권장함  
### 역할별 팀원 배분
- 잘된점: "서울 대중교통·로컬 맛집에 정통한 여행 컨시어지"로 작업 성격에 맞는 단일 전문성을 부여함  
- 보완할점: 본질상 단일 단계라 위임 부재는 감점 대상 아니나, 레포 AGENTS.md의 커넥터 전담 커니·검증 전담 베라와 연계가 전무함 — MCP 연동은 커니 프로파일 반영, 완료조건 검증은 베라 패스로  
  분리하면 로스터 정합성이 오름

#### 과제2
## 요약
### 잘된점
- 3개 섹션 정의·"3건 내외(총 7~9건)"·요약규칙(제목+2~3문장+링크1)으로 산출물이 수치로 특정됨  
- send_email.py 실행 커맨드와 플래그를 원문 그대로 명시해 실행·재현 가능성이 높음  
- "SMTP 미설정 등 발송 실패 시 HTML 저장·로그에 원인 기록"으로 실패 경로까지 완료조건화함  
### 보완할점
- "최신·중요 순으로 3건 내외"의 "중요"가 모호어 — 조회수·발행시각 등 정렬 판정기준 구체화 필요함  
- 페르소나 "루디"가 로스터 루디(DevOps 루틴 전담)와 닉네임만 같고 경력(뉴스레터 편집)이 상이 — 정합 정리 필요함  
- 3개 섹션 웹서치가 병렬 가능하나 단일 흐름에 몰림 — 섹션별 분리 수집 여지 명시 권장함## 설명  
### 구체성
- 잘된점: "출력파일: newsletter/{YYYY-MM-DD}.html", 로그에 "발송시각·수신자·섹션별 항목 수·발송 성공 여부"로 산출·완료기준이 명확함  
- 보완할점: "최신·중요 순으로 3건 내외 선별"의 '중요'가 측정 불가 — "발행 24시간 이내 우선, 동순위 시 대형 매체 우선" 등 정렬 규칙으로 바꾸면 검증 가능해짐  
### 실용성
- 잘된점: "출근 전 이메일 한 통으로 하루 트렌드를 파악"으로 WHY가 분명하고 매일 자동 루틴으로 재현 가능함  
- 보완할점: ".env의 RECIPIENTS·SMTP_*(사전 설정 필요)"만 있고 미설정 시 사전 점검 단계가 없음 — 발송 전 .env 존재·필수키 확인 스텝을 작업방법에 추가하면 실패 예방됨  
### 역할별 팀원 배분
- 잘된점: "핵심만 선별해 3분 안에 훑도록 압축"하는 큐레이션 에디터로 뉴스레터 작업에 맞는 전문성을 지정함  
- 보완할점: 루틴 작업이므로 로스터 루틴 전담 루디가 적임이나, 프롬프트의 루디 경력이 "뉴스레터 편집 10년"으로 로스터의 "DevOps 자동화·크론" 프로파일과 불일치 — 로스터 프로파일을 반영하거나 별도  
  큐레이션 역할임을 명시하면 정합성이 오름

#### 과제3
## 요약
### 잘된점
- 위험도 3단계(HIGH/MEDIUM/LOW) 정의와 "파일:라인+근거 규칙+최소 수정안 1개"로 판정기준이 촘촘함  
- 두 업무를 셀프리뷰·업무일지 2개 프롬프트로 분해해 작업 성격별로 전문성을 나눔  
- "git status에 파일 변경이 나타남" 등 실제 파일 반영을 완료조건으로 두어 정직한 보고 원칙에 부합함  
### 보완할점
- "리뷰는 읽기 전용, 수정 금지"는 명확하나 리뷰어와 업무일지 작성자 간 실행 순서·연계가 미정의됨  
- 두 페르소나 모두 로스터 6인과 무관 — 검증 전담 베라의 완료조건 확인 패스 연계 권장함  
- 셀프리뷰의 "코드 품질(가독성·네이밍·중복)" LOW 판정이 다소 주관적 — 예시 기준 보강 권장함## 설명  
### 구체성
- 잘된점: "HIGH: 병합 차단 — 보안 노출·크로스레포 수정·기능 결함" 등 등급별 정의와 "uv run ruff check {변경경로}" 실행까지 특정해 검증 가능함  
- 보완할점: 업무일지의 "기존 로그 문체를 그대로 모방"은 정성 기준 — "최근 3개 항목의 날짜·번호·레포 표기 포맷 일치"처럼 대조 항목을 명시하면 판정이 명확해짐  
### 실용성
- 잘된점: "리뷰어 지적으로 왕복하기 전에 규칙 위반을 스스로 먼저 잡고 싶음"으로 WHY가 구체적이고 칩 발급 도메인 규칙에 밀착돼 즉시 활용 가능함  
- 보완할점: 업무일지 대상 경로가 "~/workspace/work-journal(이 워크스페이스 밖)"으로 외부 의존 — 경로 부재 시 처리(생성 안내·중단)를 완료조건에 명시하면 재현성이 오름  
### 역할별 팀원 배분
- 잘된점: 리뷰는 "시니어 코드 리뷰어 12년", 로그는 "개발 생산성 엔지니어 8년"으로 작업 성격에 맞게 역할을 분리함  
- 보완할점: 두 작업 모두 단일 단계라 위임 부재는 과감점 대상 아니나, 로스터 검증 전담 베라와 무관 — 완료조건(파일 생성·git status) 확인을 베라 검증 패스로 분리하면 자가승인 위험이 줄고 로스터  
  정합성이 오름
