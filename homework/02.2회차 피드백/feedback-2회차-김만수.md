## 상세 피드백

#### 과제1
## 요약
### 잘된점
- 좌표 획득 규칙을 도구 단위로 명시(장소명 `SearchPlaceByKeywordOpen`, 주소 `geocode_address`)하여 모호성 제거함  
- 이동수단별 도구 분기와 폴백을 명확히 규정(대중교통 1순위→2순위, 자전거·도보 폴백 미적용)함  
- 파이프라인형 5인(맵핀→루트→미슐→보임→체크) 배분이 서비스 흐름과 정확히 일치함  
### 보완할점
- "경로 종점과의 도보거리(반경)"·"상위 후보 선별"에 반경 수치·개수(N) 미지정 — 정량 임계치 명시 필요함(구체성)  
- 목적지 좌표 확정 후 경로(루트)·맛집(미슐)은 병렬 가능한데 순차로만 배치 — 병렬화 여지 살릴 수 있음(역할배분)  
- 예시 링크가 `000000001` 더미라 실 API 반환 형식과의 차이 주석은 있으나 실측 샘플 1건 병기 권장함(실용성)  
## 설명
### 구체성
- 잘된점: "`hit`이면 캐시값을 사용해 재조회하지 않고, `miss`이면 실제 조회 후 `put_place`로 저장(최근 100개·7일 보관)" 등 캐시 hit/miss 판정기준이 검증 가능하게 특정됨  
- 보완할점: "경로 종점과의 도보거리(반경) 및 방문 인원을 고려해 상위 후보 선별" 문구는 반경 m·상위 N개가 없어 판정이 주관적임 → "반경 500m 이내 상위 3개" 식 수치화 필요함  
### 실용성
- 잘된점: "모든 장소명(TOP 목록·티타임·카페 등 부가 추천 포함)에 카카오맵 링크가 하이퍼링크로 걸렸는지 한 건씩 확인" 링크 누락 게이트로 결과물 신뢰성·재사용성 확보함  
- 보완할점: WHY("하나의 여정으로 통합")는 명확하나 캐시 7일 만료 후 영업시간 변동 재검증 주기가 없음 → 캐시 stale 판정 규칙 추가 시 재현성 강화됨  
### 역할별 팀원 배분
- 잘된점: "검증(체크): 소요시간·요금·좌표 정합성, 맛집 실재 여부를 재조회로 교차확인" QA(체크)를 별도 단계로 분리해 구현자 자기검증을 방지함  
- 보완할점: 좌표확정 후 경로·맛집 탐색을 단일 순차로 몰아 지연 발생 여지 → 루트·미슐 병렬 위임 후 보임이 통합하는 부분 협업형 조합 권장함

#### 과제2
## 요약
### 잘된점
- "기준 날짜 확정(필수·최우선)"에 PowerShell 조회 명령을 명시해 날짜 하드코딩을 원천 차단함  
- 완료 판정을 "종료 코드 0 + `[OK] 발송 성공` 로그 + `RESULT` JSON `ok:true`"로 3중 증거화함  
- 써치→큐레→에디→센더 4단계 파이프라인과 팀원 전문성 매핑이 정확히 대응됨  
### 보완할점
- "중요도 랭킹 후 상위 항목만 선별"에 상위 개수(N)가 없어 발송 분량이 불확정 — 임계 개수 명시 필요함(구체성)  
- 팩트체크를 큐레 단독 수행, 독립 검증 단계 없음 — 수치·전황 주장의 교차검증 주체 분리 검토 권장함(역할배분)  
- 뉴스 없을 시 "발송 또는 생략" 선택지가 양립 — 루틴 자동실행 특성상 기본 동작 1개로 확정 권장함(실용성)  
## 설명
### 구체성
- 잘된점: "`{{FOOTER_NOTE}}` = 뉴스 있음: ... / 뉴스 없음: ..." 플레이스홀더 치환값을 분기별로 완전 특정해 편집 재현성이 높음  
- 보완할점: "지역별 그룹화·중요도 랭킹 후 상위 항목만 선별" 문구에 상위 N개 기준이 없음 → "지역별 최대 3건, 총 10건 이내" 식 정량 상한 부여 필요함  
### 실용성
- 잘된점: "`templates/newsletter.html` 뼈대를 재사용함(헤더·색상·카드 구조·푸터 고정)" 디자인 일관성 고정으로 매 실행 결과가 균질해 루틴 실무에 즉시 적용 가능함  
- 보완할점: "발행일 불명 기사는 제외"는 좋으나 24시간 창 경계 시각(UTC/KST) 기준이 없어 국내외 소스 혼재 시 오차 가능 → 타임존 기준 명시 필요함  
### 역할별 팀원 배분
- 잘된점: "센더 위임 — 종료 코드·로그·JSON 결과로 성공 검증" 발송과 검증을 딜리버리 역할에 명확히 귀속시켜 가정 보고를 차단함  
- 보완할점: 큐레의 팩트체크가 곧 유일한 검증이라 오정보 리스크 집중 → 고위험(사상자 수치) 항목만 클로니 교차확인 게이트 추가 권장함

#### 과제3
## 요약
### 잘된점
- 정량 합격기준을 예시로 못박음("번들 차이 10% 이상 우위", "10k행 p95 < 100ms", "콜드스타트 p95 < 1000ms")  
- 단계별 "전이 게이트"와 "구현자↔검증자 분리(코디↔베리)"로 재현성·검증 독립성을 확보함  
- React/Vue·Supabase 두 주제에 R&D 4역할 파이프라인을 일관 재사용해 확장성이 높음  
### 보완할점
- "미지정 시 합리적 기본값 가정 후 명시"에서 기본값 실체가 없어 착수 조건이 불확정 — 기본값 표 제시 권장함(구체성)  
- 두 프롬프트 [역할] 특화 지점이 문장 수준으로 얕아 주제별 전문성 차별화 여지 있음(역할배분)  
- 산출물 md+pptx 2종을 매번 강제 — 의사결정 단계엔 md 우선, pptx는 확정 후 생성 등 단계 최적화 여지 있음(실용성)  
## 설명
### 구체성
- 잘된점: "본인 행 조회: 100% 성공 / 타인 행 조회 시도: 100% 차단(0건 반환) / 우회 시도: 100% 차단" RLS 합격기준이 이진 판정 가능하게 특정됨  
- 보완할점: "[입력정보] ... 미지정 시 합리적 기본값 가정 후 명시" 문구는 기본값 자체가 없어 재현이 실행자에 의존 → 대상 앱 유형·성능목표 등 항목별 기본값을 표로 고정 필요함  
### 실용성
- 잘된점: "재현 가능한 PoC 코드 위치 및 실행·측정 방법"과 "N회 반복 후 중앙값·p95로 보고, 버전·설정·시드 기록"으로 결과가 실무 의사결정 문서로 바로 활용됨  
- 보완할점: pptx는 "회사 템플릿 복제·렌더링 확인" 규정은 좋으나 md와 동시 강제라 초기 반복 비용 큼 → 판정 확정 후 pptx 생성 게이트 분리 시 실행 효율 향상됨  
### 역할별 팀원 배분
- 잘된점: "베리(검증·QA)는 구현자(코디)와 반드시 분리하여 위임함. 구현자 자기검증 금지"로 가짜 완료를 구조적으로 차단함  
- 보완할점: 리써니·아키의 [역할] 특화가 "조사"·"실험 설계" 수준 서술에 머묾 → React 반응성 모델 벤치 항목 등 주제별 담당 산출물을 역할에 구체 귀속 권장함
