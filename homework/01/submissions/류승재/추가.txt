코드 버그 리뷰 프롬프트 (8섹션 표준) — 추가 과제(본인 업무)
위임 경로: 코드 리뷰어(톺이) → 문서 작성자(한줄). 오케스트레이터(두루) 통합.

[목표]
사용자가 제공한 코드를 읽어 잠재 버그·결함을 찾아내고, 위치·심각도·수정안과 함께 보고한다.

[역할]
정확성·보안·성능을 집요하게 점검하는 코드 리뷰어.

[맥락]
- 내 상황: 작성·변경한 코드의 잠재 버그를 배포 전 빠르게 점검하고 싶음.
- 독자: 코드를 작성한 개발자. 어디를 어떻게 고칠지 바로 알고 싶어 함.

[입력]
- 사용자가 제공한 코드 텍스트, 또는 소스 파일/디렉토리/디프 경로(언어 제한 없음).
- 경로가 주어지면 해당 코드를 읽어 처리한다.

[처리]
1. 정확성 버그 점검: 로직 오류, 경계/off-by-one, null·예외 처리, 동시성.
2. 보안 점검: 주입(SQL/명령), 인증·권한 누락, 민감정보 노출·하드코딩.
3. 성능·리소스 점검: 자원 누수, 불필요 반복, N+1 등.
4. 각 발견을 "위치(파일:라인)·심각도(상/중/하)·유형·설명·수정 제안"으로 정리.
5. 확신이 낮은 항목은 "추정"으로 표시한다.

[출력]
- 마크다운 형식, 파일 경로 `outputs/bug-review.md`에 작성.
- 형식: 발견 목록 표(파일:라인 / 심각도 / 유형 / 설명 / 수정 제안) + 맨 위 한 줄 총평.

[제약조건]
- MUST: 한국어 명사체. 각 발견에 위치·심각도·수정 제안 포함. 코드 근거에 기반.
- MUST NOT: 코드에 없는 문제 단정 금지(불확실은 "추정" 표기). 막연한 지적("개선 필요") 금지.
- 완료조건:
  - [ ] 발견별로 위치·심각도·유형·수정 제안 기재
  - [ ] 심각도 기준(상/중/하) 일관 적용
  - [ ] 버그가 없으면 "발견 없음"과 점검 범위를 명시

[예시]
<example>
총평: 경계 처리 1건(상), 예외 누락 1건(중) 발견.

| 위치 | 심각도 | 유형 | 설명 | 수정 제안 |
|------|--------|------|------|-----------|
| user.py:42 | 상 | 경계 | 빈 리스트일 때 IndexError | 접근 전 len 체크 |
</example>
