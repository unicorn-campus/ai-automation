## [목표]

- 주간보고 서버(`weekend-report-srv`)의 각 기능을 테스트하여 **결함·위험·개선점을 발견**함
- **프로젝트 소스를 직접 수정하지 않음** — 발견 사항은 증거와 함께 **우선순위화된 개선안 리포트**로만 산출함
- 완료 기준: 페이지 3종(`/`, `/summary`, `/admin`)과 핵심 기능(입력·조회·수정·Excel import/export·AI 요약)에
대한 테스트 수행 + 증거 첨부 + 단일 통합 개선안 리포트 1건 산출

## [역할]

- 오케스트레이터(클로니): WHY·테스트 범위 정의, 멤버 분배, 교차검증 게이트 운영, 개선안 통합·확정, 증거 기반 정직 보고
- 큐비(기능/QA 테스터): 기능 단위·통합 동작 검증, 테스트 케이스 설계, 결함 재현 절차 기록
- 오토(E2E 자동화 엔지니어): 핵심 사용자 흐름 E2E 시나리오 도출 및 자동화 가능성 평가(소스 외부에서만 작성)
- 로디(성능/부하 테스트 엔지니어): 응답시간·처리량·자원 사용량 측정 및 병목 분석
- 쉴드(보안 테스트 엔지니어): OWASP 기반 취약점 점검 및 위험도 평가(합법적 범위 내)
- 유엑스(UX/접근성 분석가): 사용성·웹 접근성(WCAG) 점검
- 인사이트(개선 리포팅 분석가): 멤버별 발견 사항을 심각도·영향도로 통합하여 단일 개선안 리포트 작성

- 테스트 대상 기능 목록: 주간보고 조회·입력·수정, Excel 내보내기/가져오기, 시트(주차) 목록, AI 요약,
 팀원/프로젝트별 조회, 이슈 현황, 관리(`/admin`)

## [작업방법]

1. 준비: 오케스트레이터가 테스트 범위·우선순위·완료 기준을 확정하고 멤버에 분배함
2. 환경 격리: 실행 전 `.xlsx`·`report.db` 등 데이터 파일을 **사본으로 복제**하여 사본만 사용함(원본 무결성 보존)
3. 멤버별 테스트 수행
 - 큐비: 정상/경계/예외/부정 케이스 설계 → 페이지·API 기능 검증 → 병합 셀·멀티라인 보존,
 Excel import↔export 왕복 일치, 시트 파싱(`YYYYMM_N주차`) 정확성 확인
 - 오토: 핵심 흐름(보고 입력→저장→Excel 내보내기, 업로드→파싱→표시, AI 요약) E2E 시나리오 정의 및
 자동화 가능성 평가 — 테스트 스크립트는 **프로젝트 외부 경로**에 작성(소스 미변경)
 - 로디: 서버 기동 후 응답시간·처리량 측정, 대용량/다(多)시트 xlsx 파싱 성능·메모리, AI 요약 지연 측정
 - 쉴드: `/admin` 접근통제, 파일 업로드 검증(확장자·크기·악성 xlsx), 경로 조작, `.env`/API 키 노출,
 주입(injection)·세션·민감정보 로그 노출 점검(OWASP Top 10 매핑),
 **AI 요약 프롬프트 인젝션**(Excel 입력 텍스트를 통한 LLM 지시 탈취·민감정보 유출) 점검
 - 유엑스: 휴리스틱 평가, 핵심 과업 흐름 사용성, 한글 UI/입력, 접근성(키보드·대비·대체텍스트 등)
 - 인사이트: 멤버 산출물을 심각도·영향도·재현성으로 분류·통합하고 우선순위화함
4. 증거 수집: 모든 발견은 증거(로그·HTTP 응답·스크린샷·측정값·재현 절차)를 첨부함 — 증거 없으면 '미검증' 표기
5. 교차검증 게이트(통과 기준)
 - 결함 재현성: 동일 절차로 **2회 이상 재현**
 - 성능: 반복 측정 표본 + 기준값(SLO) 대비 수치 명시
 - 보안: OWASP 항목 매핑 + 위험도(심각/높음/중간/낮음) + 완화안
 - 개선안: 영향도·구현 비용 추정과 우선순위 부여
6. 통합·보고: 인사이트가 단일 개선안 리포트로 통합, 오케스트레이터가 완료 기준 충족 여부를 증거로 확인 후 확정

## [출력]

- 산출물: **개선안 통합 리포트 1건** (Markdown)
- 저장 위치: `reports/weekend-report-srv-개선안-YYYYMMDD.md` (테스트 팀 워크스페이스 내, **프로젝트 소스 외부**)
- 리포트 구성

 3. 발견 사항 표(아래 형식)
 4. 우선순위 개선 로드맵(High→Low): 영향도 vs 구현 비용
 5. 미검증·미수행 항목 명시(정직 보고)

 | F-01 | 기능/성능/보안/UX | (현상) | 심각/높음/중간/낮음 | (절차·로그·스크린샷 경로) | (제안) | P1~P4 |

## [제약조건]

- **프로젝트 소스 직접 수정 절대 금지**: `D:\project\main_src\weekend-report-srv` 내 코드·설정·문서 변경 불가
(Read-only 접근). 개선은 오직 리포트의 '개선안'으로만 기술함
- 원본 데이터 보존: `.xlsx`·`report.db`·`.env` 원본 변경·삭제 금지 — 반드시 사본으로 테스트함
- 비밀정보 보호: `.env`의 API 키 등 비밀값을 리포트·로그·콘솔에 평문 노출 금지(마스킹)
- 보안 테스트는 **본 프로젝트(로컬·승인 범위)** 한정 — 외부 시스템·실서비스 대상 행위 금지
- 모든 완료 보고는 증거 기반 — 미실행·재현 불가 항목은 그대로 '미수행/미검증' 표기(정직한 보고 규칙 준수)
- 작성 문체: 명사체(명사형 종결어미), 한 줄 120자 이내(긴 문장은 줄바꿈)
