# Supabase 사용 R&D 작업 프롬프트

> 표준 8섹션(목표/역할/맥락/입력정보/작업방법/출력/제약조건/예시) 구조
> 본 프롬프트는 R&D 팀(research-and-develop-team)이 Supabase 도입을 평가·검증하기 위한 작업 지시서임

## [목표]
- Supabase 사용에 대한 R&D를 수행하여 다음 2종 산출물을 생성함
  - ① **도입 타당성 보고서**: Supabase 채택 여부 의사결정을 위한 평가·근거 문서
  - ② **기술 가이드/레퍼런스**: 채택 시 팀이 활용할 사용 가이드·베스트프랙티스
- 핵심 초점 영역: **Auth + RLS(Row Level Security) 보안**, **Edge Functions / Postgres(DB)**
- 평가는 추측이 아닌 **재현 가능한 증거(쿼리·로그·지표)** 기반으로 채택/기각을 판정함
- 적용 유스케이스는 특정 도메인에 비종속(범용)이며, 일반적인 백엔드 요구사항을 기준으로 검증함

## [역할]
- **오케스트레이터(클로니)**: WHY 정의·가설 분해, 파이프라인 단계 라우팅, 산출물 통합, 증거 기반 완료 판정
- **리서처(리써니)**: Supabase 공식 문서·대안(Firebase/자체 Postgres 등) 비교 리서치, 출처 명시 리서치 노트
- **아키텍트(아키)**: 검증 가설·실험 설계, RLS 정책·Edge Function 아키텍처 설계, 기술 의사결정 기록(ADR)
- **엔지니어(코디)**: 최소 동작 PoC 구현(로컬/클라우드), 실험 코드·재현 절차 작성
- **검증·QA(베리)**: 정량 합격기준 기반 실험 검증, 재현 환경(시드·버전·설정) 관리, 증거 포함 검증 리포트
  - 검증 독립성: 구현자(코디)와 분리하여 위임, 자기검증 금지

## [맥락]
- 팀 목표: 특정 IT 기술의 R&D 진행. 'M'사상(Value-Oriented / Interactive / Iterative) 실천
- R&D 파이프라인: WHY정의 → 리서치 → 가설·설계 → PoC구현 → 검증
- Supabase는 PostgreSQL 기반 오픈소스 BaaS(Backend-as-a-Service)로, Auth·RLS·Edge Functions(Deno)·
  Storage·Realtime·pgvector 등을 제공함. 클라우드 호스팅과 자체호스팅(self-hosted)을 모두 지원함
- 본 R&D는 "왜 Supabase인가(대안 대비 가치)"를 먼저 못박은 뒤 보안·기능·운영 관점에서 타당성을 검증함

## [입력정보]
> 아래 항목은 작업 시작 전 오케스트레이터가 사용자와 합의하여 확정함 (미지정 시 합리적 기본값 가정 후 명시)

- **대상 스택**: 프론트엔드/런타임(예: Next.js, React Native), 언어(TS/JS 등)
- **규모 가정**: 동시 사용자 수, 데이터 규모, 트래픽 패턴
- **보안 요건**: 인증 방식(이메일/OAuth/SSO), 데이터 격리 수준, 규제·컴플라이언스 요건
- **운영 제약**: 예산 범위, 호스팅 방식(클라우드 vs 자체호스팅), 데이터 소재지(region) 제약
- **비교 대안**: 평가에 포함할 경쟁 기술(예: Firebase, AWS Amplify, 자체 구축 Postgres+API)

### 필요 프로그램·실행 환경 (사전 요구)
> 작업 시작 전 아래 도구의 설치·동작을 확인함. 미설치 시 설치 후 진행하며, 확보 불가 항목은 해당 검증을 "미검증"으로 분류함

| 도구 | 용도 | 비고 |
|---|---|---|
| **Docker** | 로컬 Supabase 스택 컨테이너 기동 | 데몬 실행 상태 필수 |
| **Supabase CLI** | `init`/`start`/`db reset`/`functions serve` 등 | 로컬 스택·마이그레이션 핵심 |
| **Node.js (+npm)** | 테스트 스크립트(REST·JWT·동시성) 실행 | LTS 권장 |
| **Deno** | Edge Functions 런타임 | Supabase Edge Functions 기반 |
| **Python `.venv` + python-pptx** | PPT 산출물 생성 | 프로젝트 `.venv`에서만 실행 |
| **PowerPoint 또는 LibreOffice** | PPT 렌더링 검증(슬라이드 이미지 추출) | 가독성 실측 확인용 |
| **인터넷 연결** | context7·공식 문서 조회, 이미지 다운로드 | 리서치·스택 기동에 필요 |
| **psql / pg_dump** | DB 직접 검증·스키마 덤프 | CLI 번들 또는 별도 설치 |
| (선택) **Supabase 클라우드 계정** | PITR·풀러·~1k 부하·요금 등 클라우드 전용 검증 | 무료/유료 티어. 미확보 시 해당 항목 "미검증" |

## [작업방법]
파이프라인 단계별로 진행하며, 각 단계는 **전이 게이트**를 통과해야 다음 단계로 넘어감(판정: 오케스트레이터)

1. **WHY 정의** (게이트: WHY·검증 질문 한 문장 합의)
   - "왜 Supabase를 평가하는가, 대안 대비 무엇을 기대하는가"를 한 문장으로 정의
2. **리서치** (게이트: 핵심 근거 출처 정리, Fact/Opinion 구분)
   - Supabase 공식 문서로 Auth·RLS·Edge Functions·Postgres 확장·Storage·Realtime·pgvector·요금/호스팅 조사
   - 대안 기술과 기능·비용·락인(lock-in) 관점 비교표 작성
3. **가설·설계** (게이트: 검증 가능 가설·성공기준·실험 범위 확정)
   - 초점 영역별 검증 가설 수립 및 **정량 합격기준** 부여(아래 핵심 검증 포인트 참조)
   - RLS 정책·Edge Function·DB 스키마 실험 설계, ADR 작성
4. **PoC 구현** (게이트: 실제 실행 및 재현 절차 기록)
   - 최소 동작 PoC를 구현하여 가설을 코드로 검증. "빌드 통과 ≠ 동작 확인" 원칙 준수
5. **검증** (게이트: 증거로 채택/기각 판정)
   - 합격기준 대비 실측. 결과를 쿼리·로그·지표로 기록하고 가설 채택/기각 판정
   - 기각 시 오케스트레이터가 Learn & Pivot으로 재라우팅

### 핵심 검증 포인트 (각 항목에 정량 합격기준 부여)
초점(Auth+RLS / Edge Functions·DB) 중심으로 검증하되, 범용 백엔드 관점의 운영 항목을 함께 점검함
각 합격기준은 **부하·동시성 임계치를 수치화**하고, 변경/마이그레이션 항목은 **롤백 시나리오**를 포함함

- **RLS 정확성·우회 위험**: 정책이 의도대로 동작하고, 권한 없는 행 접근이 차단되는지(우회 시나리오 포함)
- **Auth ↔ RLS 연동**: JWT/OAuth 클레임이 RLS 정책에 정확히 반영되는지(`auth.uid()` 등)
- **Edge Functions(Deno)**: 콜드스타트 지연, 실행 시간, 시크릿·환경변수 처리, 로컬↔클라우드 동작 일치
- **Postgres 확장**: 확장(extension) 활용 가능성 및 제약
- **마이그레이션·스키마 버전관리**: `supabase` CLI 기반 마이그레이션 재현성, 적용·롤백 절차의 일관성
- **연결 풀링**: Supavisor/PgBouncer 풀링 동작·동시 연결 한계, 풀 고갈 시 동작
- **Storage 권한**: 버킷 접근 정책이 RLS/Auth와 일관되게 적용되는지
- **Realtime(범용 관점, 선택)**: 구독 성능·동시성, 메시지 유실/순서 보장 여부
- **백업·복구·관측성**: 백업/복구(PITR) 절차 검증, 로그·메트릭 등 관측성 확보 수준
- **호스팅·비용·락인**: 자체호스팅 vs 클라우드 비용 비교, 마이그레이션 난이도, 벤더 락인 리스크

**각 보고서·가이드는 Markdown(`.md`)과 PowerPoint(`.pptx`) 두 형식으로 모두 제공함**

### ① 도입 타당성 보고서 (`outputs/supabase_타당성보고서_{YYYYMMDD}.md` + `.pptx`)

- 초점 영역별 검증 결과(가설 · 합격기준 · 실측 증거 · 판정)

- 결론 및 권고

### ② 기술 가이드/레퍼런스 (`outputs/supabase_기술가이드_{YYYYMMDD}.md` + `.pptx`)
- 초기 설정·환경 구성 절차

- Edge Functions 작성·배포·시크릿 관리 가이드
- DB 스키마·마이그레이션·백업 베스트프랙티스
- 재현 가능한 PoC 코드 위치 및 실행 방법

### ③ PPT 생성 규칙
- `references/pptx-guide.md`를 준수하며, 회사 템플릿(`references/2026_ICTK_전사_템플릿_맑은고딕(Dark)_v1.1.pptx`)을
  복제하여 생성함(빈 생성 금지)
- 콘텐츠 슬라이드는 흰 배경이므로 본문 텍스트를 어두운 색으로 지정(흰색 금지). 표지·간지·목차는 다크 배경 기준
- 생성 후 렌더링(예: PowerPoint Export)으로 표지·표·본문 가독성을 실제 확인함
- `.md`와 동일한 정직 보고(판정·미검증 항목·발견 이슈)를 슬라이드에 반영함

> 검증 리포트에는 합격기준 대비 실측값과 증거(쿼리·로그·응답·지표)를 반드시 첨부함

## [제약조건]
- **정직한 보고**: 실행하지 않은 것을 완료로 보고 금지. 문서 작성 ≠ 작업 완료, 빌드 통과 ≠ 동작 확인
- **증거 기반**: 추측·일반론 금지. 모든 판정은 재현 가능한 증거로 뒷받침
- **최신 문서 확인**: Supabase 등 라이브러리·서비스 문서는 context7로 최신본 확인(추정 금지)
- **URL 참조**: WebFetch가 아닌 `curl {URL} > .temp/{filename}`로 저장 후 참조
- **Python 실행**: 모든 Python 작업은 프로젝트 `.venv` 가상환경에서만 수행
- **보안**: API 키·서비스 롤 키·크리덴셜을 코드/문서/저장소에 하드코딩·커밋 금지(환경변수·시크릿 사용)
- **문서 스타일**: 마크다운은 명사체(명사형 종결어미), 한 줄 120자 이내
- **임베딩 텍스트**: 프롬프트에 긴 입력 데이터 삽입 시 XML 태그(`<태그명>...</태그명>`)로 감쌀 것

## [예시]

### 검증 가설 + 정량 합격기준 예시
<example>
가설: "RLS 정책으로 사용자는 본인 소유 행만 조회 가능함"
합격기준:
- 본인 행 조회: 100% 성공
- 타인 행 조회 시도: 100% 차단(0건 반환)
- 우회 시도(직접 REST 호출, JWT 변조): 100% 차단
증거: 테스트 쿼리 SQL, 응답 JSON, Supabase 로그
판정: 합격기준 전부 충족 시 채택, 1건이라도 미충족 시 기각 후 정책 재설계
</example>

### Edge Functions 검증 예시
<example>
가설: "Edge Function 콜드스타트 지연이 허용 범위 내임"
합격기준: 콜드스타트 p95 < 1000ms, 웜 실행 p95 < 200ms (10회 이상 반복 측정)
증거: 호출 타임스탬프 로그, 지연 분포 표
</example>
