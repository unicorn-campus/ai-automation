## [작업방법]
파이프라인 단계별로 진행하며, 각 단계는 **전이 게이트**를 통과해야 다음 단계로 넘어감(판정: 오케스트레이터)

1. **WHY 정의** (게이트: WHY·검증 질문 한 문장 합의)
   - "왜 Supabase를 평가하는가, 대안 대비 무엇을 기대하는가"를 한 문장으로 정의
2. **리서치** (게이트: 핵심 근거 출처 정리, Fact/Opinion 구분)
   - Supabase 공식 문서로 Auth·RLS·Edge Functions·Postgres 확장·Storage·Realtime·pgvector·요금/호스팅 조사
   - 대안 기술과 기능·비용·락인(lock-in) 관점 비교표 작성
3. **가설·설계** (게이트: 검증 가능 가설·성공기준·실험 범위 확정)
   - 초점 영역별 검증 가설 수립 및 **정량 합격기준** 부여(아래 핵심 검증 포인트 참조)
   - RLS 정책·Edge Function·DB 스키마 실험 설계, ADR 작성
4. **PoC 구현** (게이트: 실제 실행 및 재현 절차 기록)
   - 최소 동작 PoC를 구현하여 가설을 코드로 검증. "빌드 통과 ≠ 동작 확인" 원칙 준수
5. **검증** (게이트: 증거로 채택/기각 판정)
   - 합격기준 대비 실측. 결과를 쿼리·로그·지표로 기록하고 가설 채택/기각 판정
   - 기각 시 오케스트레이터가 Learn & Pivot으로 재라우팅

### 핵심 검증 포인트 (각 항목에 정량 합격기준 부여)
초점(Auth+RLS / Edge Functions·DB) 중심으로 검증하되, 범용 백엔드 관점의 운영 항목을 함께 점검함
각 합격기준은 **부하·동시성 임계치를 수치화**하고, 변경/마이그레이션 항목은 **롤백 시나리오**를 포함함

- **RLS 정확성·우회 위험**: 정책이 의도대로 동작하고, 권한 없는 행 접근이 차단되는지(우회 시나리오 포함)
- **Auth ↔ RLS 연동**: JWT/OAuth 클레임이 RLS 정책에 정확히 반영되는지(`auth.uid()` 등)
- **Edge Functions(Deno)**: 콜드스타트 지연, 실행 시간, 시크릿·환경변수 처리, 로컬↔클라우드 동작 일치
- **Postgres 확장**: 확장(extension) 활용 가능성 및 제약
- **마이그레이션·스키마 버전관리**: `supabase` CLI 기반 마이그레이션 재현성, 적용·롤백 절차의 일관성
- **연결 풀링**: Supavisor/PgBouncer 풀링 동작·동시 연결 한계, 풀 고갈 시 동작
- **Storage 권한**: 버킷 접근 정책이 RLS/Auth와 일관되게 적용되는지
- **Realtime(범용 관점, 선택)**: 구독 성능·동시성, 메시지 유실/순서 보장 여부
- **백업·복구·관측성**: 백업/복구(PITR) 절차 검증, 로그·메트릭 등 관측성 확보 수준
- **호스팅·비용·락인**: 자체호스팅 vs 클라우드 비용 비교, 마이그레이션 난이도, 벤더 락인 리스크

## [제약조건]
- **정직한 보고**: 실행하지 않은 것을 완료로 보고 금지. 문서 작성 ≠ 작업 완료, 빌드 통과 ≠ 동작 확인
- **증거 기반**: 추측·일반론 금지. 모든 판정은 재현 가능한 증거로 뒷받침
- **최신 문서 확인**: Supabase 등 라이브러리·서비스 문서는 context7로 최신본 확인(추정 금지)
- **URL 참조**: WebFetch가 아닌 `curl {URL} > .temp/{filename}`로 저장 후 참조
- **Python 실행**: 모든 Python 작업은 프로젝트 `.venv` 가상환경에서만 수행
- **보안**: API 키·서비스 롤 키·크리덴셜을 코드/문서/저장소에 하드코딩·커밋 금지(환경변수·시크릿 사용)
- **문서 스타일**: 마크다운은 명사체(명사형 종결어미), 한 줄 120자 이내
- **임베딩 텍스트**: 프롬프트에 긴 입력 데이터 삽입 시 XML 태그(`<태그명>...</태그명>`)로 감쌀 것

[목표]
지정한 경로 하위에 있는 gds, lef, cdl, liberty 각 디렉토리에서
버전별 하위 폴더(v0.0, v0.1, v0.2 ...)를 스캔하여 각 파일 타입별 최신 버전을

[역할]
너는 반도체 설계 데이터(PDK/라이브러리) 관리를 지원하는 셸 스크립트 작성 담당자임
버전 폴더 구조를 정확히 파싱하고, source 가능한 안정적인 스크립트를 작성함

[맥락]
설계 데이터는 파일 타입(gds, lef, cdl, liberty)별 디렉토리 아래에 버전 폴더로
관리됨. 예: {base}/gds/v0.2/cell.gds, {base}/lef/v0.3/cell.lef
타입마다 존재하는 최신 버전이 다를 수 있어(gds는 v0.2까지, lef는 v0.3까지 등),
각 타입의 최신 버전을 한눈에 파악하는 것이 목적임

[입력정보]
- 대상 베이스 경로: 사용자가 지정(스크립트 인자 또는 변수)
- 하위 구조: {base}/{type}/v{major}.{minor}/... 형태
- 대상 타입: gds, lef, cdl, liberty

[작업방법]
1. 대상 베이스 경로 아래에서 gds, lef, cdl, liberty 디렉토리 각각 탐색
2. 각 타입 디렉토리 안의 버전 폴더명(v로 시작) 목록 수집
3. 버전 문자열을 숫자 기준으로 정렬(v0.10 > v0.2 가 되도록 major.minor 수치 비교)
4. 각 타입별 최신 버전 1개 선정
5. 결과를 타입 / 최신버전 / 전체버전목록 컬럼의 표로 터미널 출력
6. 디렉토리나 버전 폴더가 없는 타입은 "없음"으로 표기

[출력]
- 실행 가능한 bash 스크립트 파일: results/find-latest-version.sh
- 스크립트 실행(source 또는 실행) 시 터미널에 아래 형태의 표 출력:
 +----------+-------------+---------------------------+
 | TYPE | LATEST | ALL VERSIONS |
 +----------+-------------+---------------------------+
 | gds | v0.2 | v0.0, v0.1, v0.2 |
 | lef | v0.3 | v0.0, v0.1, v0.2, v0.3 |
 | cdl | v0.1 | v0.0, v0.1 |

### 참고: gds, lef, cdl, liberty 비유로 쉽게 이해하기

반도체 셀 설계를 아파트 건축에 비유하면 네 파일 타입은 다음 역할을 담당함

- **GDS**: 아파트 실제 시공 도면. 마감재·배관까지 다 그려져 공장(파운드리)이 그대로 찍어내는 최종 도면(예: `cell.gds`)
- **LEF**: 아파트 배치도. 크기·출입구(핀) 위치 등 요약 정보만 담아 여러 세대(셀)를 단지(칩)에 배치할 때 사용(예: `cell.lef`)
- **CDL**: 아파트 전기 배선 계통도. 트랜지스터 간 연결 관계를 나타내며 시공 도면(GDS)과 배선이 일치하는지  
  대조(LVS 검증)하는 데 사용(예: `cell.cdl`)
- **Liberty(.lib)**: 가전제품 스펙시트. 신호 지연시간·전력 소모량 등 성능 수치를 정리한 표(예: `cell.lib`)

즉 같은 셀 하나도 실물 도면(GDS)·배치 요약(LEF)·배선 연결정보(CDL)·성능 스펙(Liberty)이 각각 별도 파일로 관리되며,  
파일 타입마다 버전이 따로 올라가기 때문에 김승호님 프롬프트처럼 타입별 최신 버전을 한눈에 확인할 필요가 생김

### 참고: 최종현님 프롬프트 비전문가용 해설

#### 한 줄 요약
- 보안칩(스마트카드·인증칩 등)에 들어가는 소프트웨어를 **공장 출하 전에 보안 전문가 4명이 나눠 검사**하고,  
  발견한 문제를 심각도·수리 방안과 함께 리포트로 정리하는 작업 지시서임
- 비유: 자동차 출고 전 안전검사와 유사함. 브레이크·에어백·엔진을 각 분야 검사관이 점검하고,  
  결함을 "심각도 등급 + 수리 권고"로 보고서에 정리하는 것과 같은 구조임

#### 이 프롬프트가 다루는 대상
- **보안 IC**: 비밀번호·암호키 같은 민감정보를 안전하게 보관·처리하도록 설계된 특수 반도체 칩  
  (예: 신용카드 IC칩, 여권 전자칩, 자동차 보안모듈)
- **펌웨어**: 칩 안에 내장되어 칩을 동작시키는 소프트웨어. 가전제품 속 "붙박이 프로그램"에 해당함
- **암호 구현 코드**: 데이터를 암호화·복호화하고 서명을 검증하는 프로그램 부분

#### 핵심 용어 쉽게 풀기 (검사 항목)
- **키 관리 / 키 라이프사이클**: 암호키(=금고 비밀번호)가 만들어지고 쓰이고 폐기되는 전 과정.  
  금고 열쇠를 만들고 쓰고 버리는 절차가 안전한지 보는 것과 같음
- **평문 노출**: 암호키가 암호화되지 않은 맨 상태로 메모리에 남아 새어나가는 문제.  
  비유: 금고 비밀번호를 포스트잇에 적어 책상에 붙여둔 상태
- **메모리 잔존 / 미소거**: 다 쓴 비밀정보를 메모리에서 지우지 않아 나중에 훔쳐볼 수 있는 문제.  
  비유: 복사기에 원본 서류를 두고 나오는 것. 대응책이 **secure zeroization**(사용 직후 확실히 덮어써 지움)
- **난수 / RNG / 엔트로피**: 예측 불가능한 무작위 숫자를 만드는 기능. 암호의 안전성은 "얼마나 진짜 무작위인가"에 달림.  
  비유: 주사위가 특정 숫자만 잘 나오면 도박판이 조작되듯, 난수가 편향되면 암호가 뚫림.  
  **DRBG**는 난수 생성기의 한 종류, **헬스테스트**는 난수기가 고장났는지 자가진단하는 절차, **재시드**는 새 무작위 재료를 다시 넣는 것
- **상수시간(constant-time) 위반**: 비밀값에 따라 계산 시간이 달라지면, 그 시간차를 재서 비밀을 알아내는 공격이 가능함.  
  비유: 금고 다이얼을 돌릴 때 맞는 숫자에서만 "딸깍" 소리가 나면, 소리로 비밀번호를 유추하는 것과 같음
- **부채널(side-channel)**: 코드 자체가 아니라 **전력 소모·시간·전자파 같은 부수 신호**로 비밀을 훔치는 공격.  
  비유: 벽에 귀를 대고 옆방 대화를 엿듣는 것
- **시큐어부트(secure boot)**: 칩이 켜질 때 정품·정상 소프트웨어만 실행되도록 서명을 확인하는 절차.  
  비유: 건물 입장 시 정상 신분증만 통과시키는 출입 게이트. **서명검증**은 그 신분증 진위 확인 단계
- **메모리 안전 결함**: 프로그램이 허용된 메모리 범위를 벗어나 오작동·해킹 통로가 되는 문제
  - **버퍼 오버플로**: 정해진 상자보다 많은 짐을 넣어 옆 상자를 망가뜨림
  - **정수 오버플로**: 계산 숫자가 담을 수 있는 최대값을 넘어 엉뚱한 값으로 뒤집힘 (자동차 주행거리계가 999999 넘어 0이 되는 것)
  - **use-after-free**: 이미 반납·폐기한 메모리를 계속 쓰는 문제 (반납한 사물함을 남이 쓰는데 계속 열어보는 것)
  - **동시성**: 여러 작업이 동시에 실행될 때 순서가 꼬여 생기는 오류

#### 등장하는 기술·표준
- **PUF(Physically Unclonable Function)**: 칩마다 제조 시 생기는 미세한 물리적 차이를 "지문"처럼 활용해  
  칩 고유의 값을 만드는 기술. 복제가 불가능해 위조 방지에 쓰임. 비유: 사람마다 다른 지문·홍채
- **PQC(양자내성암호, Post-Quantum Cryptography)**: 미래의 양자컴퓨터로도 못 깨는 차세대 암호.  
  현재 암호가 양자컴퓨터에 뚫릴 위험에 대비한 것. **ML-KEM·ML-DSA·SLH-DSA**는 미국 표준(NIST)으로 채택된 PQC 알고리즘 이름들
  - **ML-KEM**: 안전하게 암호키를 주고받는 방식(키캡슐화). **디캡슐레이션**은 받은 것을 풀어 키를 꺼내는 단계
  - **ML-DSA / SLH-DSA**: 전자서명 방식. 문서가 위·변조되지 않았음을 증명하는 인감도장에 해당함
- **FIPS 140-3**: 미국 정부의 암호모듈 보안 인증 규격. 레벨이 높을수록 요구 보안 수준이 강함
- **Common Criteria(CC) / EAL**: 국제 보안 평가 표준. **EAL**은 그 등급(EAL1~7)으로, 숫자가 클수록 검증이 엄격함.  
  비유: 두 표준 모두 "이 칩은 국제 공인 보안 인증을 통과했다"는 품질 보증서에 해당함

- 보안칩은 **출하 후 결함이 발견되면 시장 회수·재인증 비용이 막대**하므로 배포 전 검사가 필수임  
  (비유: 자동차 리콜처럼 이미 팔린 제품을 되돌리면 손실이 큼)
- 그래서 4명의 전문가(키 담당·부채널 담당·메모리 담당·통합 담당)가 **서로 교차 검증**하고,
